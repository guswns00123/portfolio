## 1. 프로젝트 개요
- **한 줄 요약**: TFT 챌린저 유저 주간 경기 데이터 파이프라인 및 대시보드 분석
- **동기/배경**: 왜 이걸 만들었는지 (개인적 호기심, 기술 학습, 문제 해결 등)
TFT 게임에 관심이 많았고 이거에 대한 유저 데이터를 한번 내가 스스로 Airflow랑 파이썬 배운 기술들을 활용하여 데이터 파이프라인을 내 입맛대로 구축해보고싶어서 시작하게됨.

## 4. 핵심 구현 내용

**Problem : 경기 결과 데이터에 한번씩 데이터가 누락되는 경우가 생김**

**Cause : TFT API의 콜 제한이 있어서 스케줄링할 때 API 콜이 한번씩 제한되는 경우가 있음** 

**Solution : Lambda 함수 Trigger를 할때 동시에 여러개의 람다 함수를 Trigger하지 않고 3개씩 task를 나누어 Call하여 API 콜 제한에 넘지 않게 조정**
- **트러블슈팅**: 개발 중 겪은 문제와 해결 과정 (이게 가장 중요!)

- **성능 최적화**: 속도/메모리/비용 개선한 부분이 있다면
Airflow 자체에서 api를 호출할때 컴퓨터성능상 Airflow가 자주 뻗는 문제가 생겨 aws lambda 와 s3를 연계하여 대체하는 기술을 채택

Apache Airflow (데이터 오케스트레이션):

도입 이유: 단순 OS 크론탭(Cron) 대신 도입. 데이터의 종류(캐릭터, 경매장, 거래소)에 따라 수집 주기가 다르고, API Rate Limit으로 인한 실패 시 재시도(Retry) 로직이 필수적이었습니다. Airflow의 DAG를 활용해 복잡한 의존성을 관리하고, 파이프라인 실패 시 즉각적인 모니터링이 가능하도록 구성했습니다.

PostgreSQL + pgvector (데이터 저장소):

도입 이유: LLM 챗봇을 위한 벡터 데이터(임베딩)와 게임의 메타 데이터(관계형 데이터)를 별도의 DB(예: MySQL + Pinecone 조합)로 분리해서 관리하면 인프라 관리 리소스와 데이터 정합성 유지 비용이 두 배로 듭니다. PostgreSQL의 pgvector 익스텐션을 활용하여 RDB와 Vector DB를 단일 저장소로 통합, 시스템 복잡도를 크게 낮추었습니다.

FastAPI (AI 백엔드 서버):

도입 이유: LLM API(OpenAI) 호출과 DB 쿼리(Text-to-SQL)는 대표적인 I/O Bound 작업입니다. FastAPI의 강력한 비동기(Asynchronous) 처리 지원을 통해 다수의 사용자 챗봇 질의가 동시에 들어와도 블로킹(Blocking) 없이 병렬로 빠르게 처리할 수 있도록 설계했습니다.


3. 데이터 수집 및 설계
데이터 소스 및 Rate Limit 최적화 전략:

소스: 로스트아크 Open API

API Rate Limit 우회 및 분산 처리: API Key 1개당 '1분당 100회'라는 엄격한 수집 제한(Rate Limit)을 준수하기 위해 2개의 API Key를 목적별로 논리적 분리 적용. (Key A: 캐릭터 정보 수집 전용 / Key B: 경매장 및 거래소 데이터 수집 전용) 이를 통해 수집 병목을 해소하고 안정적인 ETL 파이프라인을 구축했습니다.

데이터 갱신 주기 (Freshness) 차등화 설계:

데이터의 특성과 변동성에 따라 배치(Batch) 주기를 세분화하여 리소스 낭비를 방지했습니다.

캐릭터 정보: 1시간 주기 기본 수집 + 유저 챗봇 질의 시 최신 정보 On-demand 수집

경매장: 일반 데이터 1시간 주기 / 가격 변동성이 높은 주요 아이템은 5분 주기(Near Real-time) 수집

거래소: 데이터 변동 폭에 따라 1일 1회 ~ 1시간 주기 혼용

데이터 볼륨:

경매장/거래소 데이터 기준 일일 약 5,000건의 트랜잭션 데이터 적재. 캐릭터 데이터는 사용자의 챗봇 활용도에 비례하여 동적으로 유기적인 데이터 축적이 이루어지는 구조로 설계.

스키마/테이블 설계: ERD 간략 소개 및 정규화/비정규화 고민의 흔적
