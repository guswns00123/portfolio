## 1. 프로젝트 개요

**게임 무물봇**('게임에 대해 **무**엇이든지 **물**어보세요'의 줄임말)은 유저들이 게임 정보를 얻기 위해 여러 외부 사이트를 옮겨 다녀야 하는 **데이터 파편화 문제**를 해결하는 AI 챗봇입니다. 정적 표 형태가 아닌, 자연어 질문 → LLM이 즉시 데이터를 추출·가공해 답변하는 **RAG(Retrieval-Augmented Generation) 기반 대화형 데이터 탐색 환경**을 구축했습니다.

**진행 현황**: 로스트아크 적용 완료 / TFT API 확장 개발 중. 타 게임 추가 확장을 위한 도메인 분리형 구조로 설계.

## 2. 나의 역할

2인 팀 중 **데이터 엔지니어링 · AI 데이터 아키텍처 영역 100% 전담**

- 외부 게임 API 수집·정제, Airflow 자동화
- PostgreSQL + pgvector 기반 RDB / Vector DB 통합 설계
- **RAG 아키텍처 설계**: Retrieval(pgvector) + Augmented(스키마 / Few-shot) + Generation(OpenAI)
- LLM 호출 모니터링 페이지 자체 개발 (팀원 프롬프트 디버깅 지원)

## 4. 데이터 설계

### 4-1. 수집 전략

**API Rate Limit 분산** — 로스트아크 API는 Key당 1분 100회 제한. 2개 Key를 목적별 분리:

| API Key | 용도 |
|---------|------|
| Key A | 캐릭터 정보 (프로필 / 장비 / 스킬) |
| Key B | 경매장 · 거래소 (거래 / 시세) |

**갱신 주기 차등화** — 데이터 변동성에 따라 배치 주기를 분리해 리소스 낭비 방지:

| 데이터 유형 | 수집 주기 | 근거 |
|------------|----------|------|
| 캐릭터 정보 | 1시간 + On-demand | 챗봇 질의 시 최신 정보 즉시 수집 |
| 경매장 (주요 아이템) | **5분** (Near Real-time) | 가격 변동성 높음 |
| 경매장 (일반) / 거래소 | 1시간 ~ 1일 | 변동 폭에 따라 혼용 |

**볼륨**: 일일 약 **5,000건** 트랜잭션 적재. 캐릭터 데이터는 챗봇 활용도에 비례해 동적 누적.

### 4-2. 스키마 구조

2개 스키마 / 5개 그룹 — `public`(챗봇 세션) + `lostark`(캐릭터 스냅샷, 정적 메타, 시장 데이터, RAG / Vector).

**핵심 설계 결정**

- **스냅샷 누적 (`(character_name, collected_at)` 복합 PK)**: `UPDATE` 덮어쓰기 대신 시점별 이력 보존 → "어제와 오늘 장비 비교" 같은 시계열 질의 가능
- **`character_info_tb` 허브 구조**: 15개 수집 테이블이 단일 마스터를 참조 → 캐릭터 추가 / 삭제 시 파급 범위 최소화

## 5. LLM 모니터링 대시보드 자체 개발

![게임 무물봇 LLM 모니터링 대시보드](/images/game-chatbot-monitoring.png)

**배경** — 챗봇은 **3개의 LLM Generator**로 답변을 생성합니다.

| Generator | 역할 |
|-----------|------|
| **Analysis** | 자연어 질문 → 핵심 키워드 / 의도 정리 |
| **SQL** | 정리된 의도 → PostgreSQL 쿼리 생성 |
| **Answer** | 쿼리 결과 → 사용자 친화 자연어 답변 |

초기에는 LangChain(LangSmith) 기반 모니터링을 썼지만, **3개 Generator의 토큰 / 응답시간 / 에러율을 단계별로 비교**하기엔 UI가 불편했고, 프롬프트를 담당하는 팀원의 디버깅·반복 개선 속도가 떨어졌습니다.

**해결** — 팀 워크플로우에 맞춘 자체 대시보드 구축:

- **Generator별 통계**: 호출수 / Input·Output 토큰 / 평균 응답 / 비용 / 에러율 한눈에 비교
- **시간대별 호출량 차트**, **호출 로그(전체 / analysis / sql / answer 필터)**, **실시간 새로고침**

**효과**: 어떤 Generator가 토큰을 가장 많이 쓰는지(예: Answer 5.1s, 63.6K input), 어떤 단계에서 에러가 나는지 즉시 식별 → 팀원 프롬프트 개선 사이클 단축, Generator별 비용 가시화로 최적화 우선순위 결정.

### [카테고리 1] RAG 및 LLM 파이프라인 고도화

<details>
<summary><strong>Point 1.</strong> RDB + Vector DB 통합 (pgvector) — <em>Cold Start 2,520ms 제거 · 토큰 84% 절감</em></summary>

**문제 상황**

자연어 질문을 SQL로 변환(Text-to-SQL)하는 과정에서 정확도를 높이기 위해 전체 데이터베이스 스키마를 프롬프트에 주입할 경우 토큰 비용이 급증했습니다. 이를 완화하고자 In-memory Vector Search를 도입했으나, GCP Cloud Run 환경 특성상 컨테이너 기동 시마다 약 5초 이상의 Cold Start 지연이 발생하는 새로운 병목에 직면했습니다.

**핵심 과제**

비용 절감과 초기 응답 지연(Cold Start) 방어를 동시에 만족하는 최적화된 **RAG Retrieval 아키텍처** 구축.

**해결 과정 — Retrieval 아키텍처 4단계 고도화**

| 단계 | 전략 | 해결된 문제 | 발생한 한계점 |
|------|------|------------|---------------|
| Phase 1 | 전체 스키마 주입 | — | 과도한 토큰 비용 발생 |
| Phase 2 | LLM 자체 판단 | 비용 절감 | 할루시네이션 (잘못된 테이블 매핑) |
| Phase 3 | In-memory Vector Search | 정확도 확보 및 비용 절감 | Cold Start 지연 (Cloud Run) |
| Phase 4 | DB-native Vector (pgvector) | **모든 문제 해결** | — |

**측정 결과**

| Phase | 총 응답시간 | 스키마 선택 소요시간 | 소모 토큰 (대비) | 정확도 |
|-------|------------|---------------------|----------------|--------|
| Phase 1 | 2,628ms | 0ms | 5,847 (기준점) | 83% |
| Phase 2 | 3,516ms (+34%) | 1,613ms | 469 (-92%) | 83% |
| Phase 3 | 2,837ms (+8%) | 898ms | 1,130 (-81%) | **100%** |
| Phase 4 | 3,147ms (+20%) | 1,003ms | 923 (**-84%**) | **100%** |

**결과**

GCP Cloud Run 환경에서 기존 In-memory Vector 방식은 컨테이너 기동 시마다 임베딩 데이터를 애플리케이션 메모리에 로드해야 했고, 이로 인해 약 **2,520ms의 초기 로드 지연**이 발생했습니다.

이를 해결하기 위해 pgvector를 도입해 **RDB와 Vector 인덱스를 단일 PostgreSQL 스토리지로 통합**했습니다. 그 결과, 애플리케이션 단의 **벡터 인덱스 로드 과정 자체를 제거**하여 초기 응답 지연 문제를 근본적으로 방어할 수 있었습니다. 또한, 메타데이터 필터링과 벡터 유사도 검색을 하나의 데이터베이스에서 처리하게 되면서 **아키텍처의 복잡도를 낮추고 관리 포인트를 일원화**할 수 있었습니다.

</details>

<details>
<summary><strong>Point 2.</strong> Few-shot 도입 및 프롬프팅 최적화</summary>

**문제 상황**

Text-to-SQL 파이프라인에서 데이터베이스 스키마만 제공하여 쿼리를 생성할 경우, 두 가지 치명적인 한계가 발생했습니다. 첫째, 사용자의 자연어 표현이 미세하게 달라질 때마다 생성되는 SQL의 구조가 흔들리는 **'재현성 부족'** 문제가 있었습니다. 둘째, 존재하지 않는 테이블이나 컬럼을 참조하는 **할루시네이션** 이 발생해 동일한 질문에도 답변 품질의 편차가 컸습니다.

**핵심 과제**

LLM의 확률적 생성 결과를 제어하여, 일관되고 예측 가능한 SQL 쿼리 생성을 보장하는 **프롬프트 아키텍처** 구축.

**해결 과정 — 두 가지 메커니즘의 결합 및 분리**

질문의 의도를 파악하는 과정과 SQL의 뼈대를 잡는 과정을 분리하여 각각 다른 방식으로 통제했습니다.

- **Few-shot 예시 주입 (재현성 및 구조 안정성 확보)**: 도메인에 특화된 '질문-SQL' 예시 쌍을 프롬프트에 동적으로 주입하여, LLM이 출력해야 할 SQL의 패턴과 조인(Join) 구조를 강제했습니다.
- **`keywords` 메타 컬럼 도입 (표현 변화에 대한 강건성 확보)**: 유저들이 자주 사용하는 게임 내 약어나 동의어(예: 뭉가, 2렙 등)를 실제 데이터베이스 값과 매핑하는 메타데이터 컬럼을 별도로 구축하여, 다채로운 자연어 표현에도 정확한 조건절(`WHERE`)을 생성하도록 유도했습니다.

**측정 결과 1 — 재현성** (동일 질문 5회 반복 테스트)

테스트 질의: *"뭉가 2레벨 효과가 어떻게 돼?"*

| 측정 항목 | Few-shot 적용 전 (Zero-shot) | Few-shot 적용 후 |
|----------|------------------------------|-----------------|
| SQL 구조 일치율 | 2/5 | **4/5** (1회 컬럼 누락 제외 일치) |
| 할루시네이션 발생 | 3/5 (없는 테이블 참조) | **0/5** (발생 없음) |
| 운영 영향도 | 간헐적으로 답변 생성 실패 | 안정적인 정상 응답 반환 |

**측정 결과 2 — 생성 안정성** (유사 표현 변형 테스트)

테스트 질의 변형: *"뭉가 2단계", "레벨2가 뭐야?", "2렙 효과"*

| 측정 항목 | Few-shot 적용 전 | Few-shot 적용 후 |
|----------|-----------------|-----------------|
| 정상 SQL 생성 성공률 | 2/3 (표현마다 쿼리 형태 변동) | **3/3** (모두 동일한 패턴으로 생성) |

**결과**

단순한 스키마 주입만으로는 확률적 모델인 LLM을 실제 서비스 레벨에서 통제하기 어렵다는 것을 확인했습니다. Few-shot 프롬프팅을 통해 쿼리의 **'구조적 뼈대'** 를 잡아주고, 별도의 메타데이터 컬럼으로 **'도메인 어휘'** 를 보정해 주는 **투트랙 전략**을 적용했습니다. 이를 통해 자연어 입력의 변동성에도 흔들리지 않는 재현성을 확보했으며, 할루시네이션을 억제하여 챗봇 응답의 신뢰도를 크게 향상할 수 있었습니다.

</details>

### [카테고리 2] 데이터 수집 및 적재 효율화

<details>
<summary><strong>Point 3.</strong> 정적 / 동적 데이터 분리 — <em>단일 페이로드 79% 절감 (82KB → 17KB)</em></summary>

**문제 상황**

외부 게임 API 응답 데이터에는 캐릭터의 장비 세팅(동적 데이터)뿐만 아니라, 스킬 상세 텍스트나 트라이포드 효과 등 변동이 거의 없는 메타데이터(정적 데이터)가 **매 수집 주기마다 중복으로 포함되어 반환**되었습니다. 이로 인해 파이프라인 구동 시 불필요한 네트워크 트래픽이 발생하고, RDB 스토리지가 빠르게 낭비되는 구조적 비효율이 존재했습니다.

**핵심 과제**

데이터의 **변동성을 기준으로 파이프라인을 분리**하여, API Rate Limit을 방어하고 스토리지 적재 비용을 최소화.

**해결 과정 — 수집 주기 분리 및 DAG 이원화**

데이터의 성격에 따라 Airflow 수집 파이프라인(DAG)을 완전히 분리했습니다.

- **정적 데이터 (메타데이터)**: API와 웹 스크래핑을 병행하여 최초 1회 일괄 적재하고, 게임 패치가 있는 시점에만 **수동으로 트리거** 하여 갱신하도록 변경했습니다.
- **동적 데이터 (캐릭터 세팅)**: 챗봇 질의에 필요한 핵심 변동 데이터만 발라내어, 짧은 주기로 **경량화된 수집**만 수행하도록 개편했습니다.

**측정 결과** (부하 시뮬레이션 기반 개선 지표)

> *현재 시스템은 챗봇 활용도에 비례해 데이터를 동적 누적하고 있으나, 아래 지표는 향후 서비스 확장에 대비하여 **'활성 캐릭터 1,000명 / 1시간 주기 수집'** 을 가정한 스트레스 시뮬레이션 수치입니다.*

| 측정 항목 | 최적화 전 | 최적화 후 |
|----------|----------|----------|
| 단일 캐릭터 수집 페이로드 | 82KB | **17KB (79% 절감)** |
| 일일 스토리지 누적량 (예상) | 약 1.9GB | **약 400MB (79% 절감)** |
| 정적 데이터 수집 빈도 | 매시간 (일 24회 중복) | 게임 패치 시 1회 |
| 파이프라인 운영 부하 | 매 주기 무거운 텍스트 파싱 발생 | 경량 트랜잭션 적재로 리소스 최소화 |

**결과**

무비판적으로 API 응답 전체를 적재하던 방식에서 벗어나, 데이터의 도메인 특성을 분석하여 수집 아키텍처를 재설계했습니다. 그 결과 **단일 페이로드 크기를 79% 절감**했습니다. 비록 현재 단계에서는 실제 일일 적재량이 크지 않지만, 이러한 선제적인 데이터 경량화를 통해 향후 트래픽이 급증하더라도 외부 API의 **Rate Limit에 걸리지 않고 스토리지 낭비를 방어할 수 있는 견고한 확장성** 을 확보했습니다.

</details>

<details>
<summary><strong>Point 4.</strong> 비정형 데이터(JSONB) 정규화 — <em>블록 I/O 33% 절감 · LLM 토큰 효율 동시 개선</em></summary>

**문제 상황**

원본 게임 API에서 제공하는 핵심 데이터(`tooltip`)가 인게임 UI 렌더링을 위한 HTML 태그(예: `<FONT>`)가 포함된 **복잡한 중첩 JSONB 형태**로 제공되었습니다. 이를 그대로 적재할 경우, LLM이 Text-to-SQL을 수행할 때 불필요한 비정형 텍스트 파싱을 유발하여 응답 속도가 저하되고 프롬프트 토큰이 낭비되며 할루시네이션이 증가하는 문제가 발생했습니다.

**핵심 과제**

비정형 API 응답 데이터를 관계형 DB 구조에 맞게 정제하여, **데이터 조회 성능을 높이고 LLM의 데이터 해석 비용을 최소화**.

**해결 과정 — ETL Transform 단계의 데이터 평탄화**

- **노이즈 제거 및 타입 캐스팅**: Airflow의 Transform 단계에서 **정규표현식** 을 활용해 불필요한 HTML 태그를 모두 제거하고, 중첩된 JSON 구조를 평탄화하여 핵심 수치만 추출해 **Numeric 타입으로 변환**했습니다.
- **스키마 직관성 확보**: 비정형 텍스트 연산을 순수 수치 연산 구조로 개편하면서, 기존에 불필요하게 존재하던 파생(총합) 테이블들을 제거하고 조인 구조를 단순화했습니다.

**측정 결과** (500건 Full Scan 기준, `EXPLAIN ANALYZE` 프로파일링)

| 측정 항목 | 최적화 전 (비정형 JSONB) | 최적화 후 (정규화된 Numeric) |
|----------|------------------------|----------------------------|
| 메모리 I/O 블록 수 | 36 블록 | **24 블록 (33% 절감)** |
| 실행 속도 | 0.644ms | 0.121ms (5.3배 향상) |
| 운영 영향도 | 파싱 오버헤드 및 토큰 낭비 | 즉각 연산 및 토큰 사용 최적화 |

**결과**

현재 500건 수준의 데이터 풀에서는 절대적인 쿼리 실행 시간의 차이(0.5ms)가 사용자 경험에 미치는 영향은 미미할 수 있습니다. 하지만, 향후 서비스가 확장되어 데이터가 **수만에서 수십만 건으로 누적되었을 때 발생할 수 있는 대규모 Full Scan 병목을 사전에 차단**하기 위해 아키텍처를 선제적으로 최적화했습니다. 결과적으로 **디스크/메모리 I/O 블록을 33% 절감**했을 뿐만 아니라, LLM이 불필요한 텍스트를 해석하는 데 낭비하던 **토큰 비용까지 동시에 해결**한 의미 있는 개선이었습니다.

</details>

## 7. 성과 및 회고

**결과물**: [챗봇 데모](https://game.mumulbot.com/)

**배운 점**

- pgvector를 활용한 **RDB + Vector DB 통합 RAG Retrieval** 설계 경험
- **RAG / Text-to-SQL 파이프라인**을 데이터 엔지니어 관점에서 직접 구축 — Retrieval 품질이 정확도·비용을 결정한다는 점 체득
- Few-shot이 단순 프롬프트 기법이 아닌, **운영 안정성을 보장하는 메커니즘**임을 측정으로 검증
- **자체 모니터링 도구 개발의 가치** — 외부 도구가 줄 수 없는 도메인 맞춤 인사이트와 빠른 디버깅 사이클

**넥스트 스텝**

- TFT API 확장 안정화 — 멀티 도메인에서 RAG가 게임 컨텍스트를 정확히 라우팅하도록 검증
- BackgroundTasks 한계 도달 시점에 Redis / Celery 점진 마이그레이션
