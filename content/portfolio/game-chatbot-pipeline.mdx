## 프로젝트 개요

**게임 무물봇**('게임에 대해 **무**엇이든지 **물**어보세요'의 줄임말)은 유저들이 게임 정보를 얻기 위해 여러 외부 사이트를 옮겨 다녀야 하는 **데이터 파편화(Fragmentation) 문제**를 해결하는 AI 챗봇입니다. 정적 표 형태가 아닌, 자연어 질문 → LLM이 즉시 데이터를 추출·가공해 답변하는 **RAG(Retrieval-Augmented Generation) 기반 대화형 데이터 탐색 환경**을 구축했습니다.

**진행 현황**: ✅ 로스트아크 적용 완료 / 🚧 TFT API 확장 개발 중 (멀티 도메인 검증 단계). 타 게임 추가 확장을 위한 도메인 분리형 구조로 설계.

### 나의 역할

2인 팀 중 **데이터 엔지니어링 · AI 데이터 아키텍처 영역 100% 전담**

- 외부 게임 API 수집·정제 (Ingestion + Transform), Airflow 자동화
- PostgreSQL + pgvector 기반 RDB / Vector DB 통합 설계
- **RAG 아키텍처 설계**: Retrieval(pgvector) + Augmented(스키마 / Few-shot) + Generation(OpenAI)
- LLM 호출 모니터링 페이지 자체 개발 (팀원 프롬프트 디버깅 지원)

## 데이터 수집 설계

**API Rate Limit 분산** — 로스트아크 API는 Key당 1분 100회 제한. 2개 Key를 목적별 분리:

| API Key | 용도 |
|---------|------|
| Key A | 캐릭터 정보 (프로필/장비/스킬) |
| Key B | 경매장·거래소 (거래/시세) |

**갱신 주기 차등화** — 데이터 변동성에 따라 배치 주기를 분리해 리소스 낭비 방지:

| 데이터 유형 | 수집 주기 | 근거 |
|------------|----------|------|
| 캐릭터 정보 | 1시간 + On-demand | 챗봇 질의 시 최신 정보 즉시 수집 |
| 경매장 (주요 아이템) | **5분** (Near Real-time) | 가격 변동성 높음 |
| 경매장 (일반) / 거래소 | 1시간 ~ 1일 | 변동 폭에 따라 혼용 |

**볼륨**: 일일 약 **5,000건** 트랜잭션 적재. 캐릭터 데이터는 챗봇 활용도에 비례해 동적 누적.

- **`character_info_tb` 허브 구조**: 15개 수집 테이블이 단일 마스터를 참조 → 캐릭터 추가/삭제 시 파급 범위 최소화
- **가격 크로스 참조**: 캐릭터 스냅샷(API)과 시장 데이터(시세 수집)는 출처가 다르지만 `name` 컬럼으로 연결 → "내 각인 시세는?", "내 장신구 동급 매물가?" 같은 질의 지원

## LLM 모니터링 대시보드 자체 개발

![게임 무물봇 LLM 모니터링 대시보드](/images/game-chatbot-monitoring.png)

**배경** — 챗봇은 **3개의 LLM Generator**로 답변을 생성합니다.

| Generator | 역할 |
|-----------|------|
| **Analysis** | 자연어 질문 → 핵심 키워드 / 의도 정리 |
| **SQL** | 정리된 의도 → PostgreSQL 쿼리 생성 |
| **Answer** | 쿼리 결과 → 사용자 친화 자연어 답변 |

초기에는 LangChain(LangSmith) 기반 모니터링을 썼지만, **3개 Generator의 토큰 / 응답시간 / 에러율을 단계별로 비교**하기엔 UI가 불편했고, 프롬프트를 담당하는 팀원의 디버깅·반복 개선 속도가 떨어졌습니다.

**해결** — 팀 워크플로우에 맞춘 자체 대시보드 구축:

- **Generator별 통계**: 호출수 / Input·Output 토큰 / 평균 응답 / 비용 / 에러율 한눈에 비교
- **시간대별 호출량 차트**, **호출 로그(전체/analysis/sql/answer 필터)**, **실시간 새로고침**

**효과**: 어떤 Generator가 토큰을 가장 많이 쓰는지(예: Answer 5.1s, 63.6K input), 어떤 단계에서 에러가 나는지 즉시 식별 → 팀원 프롬프트 개선 사이클 단축, Generator별 비용 가시화로 최적화 우선순위 결정.

## 핵심 구현 및 문제 해결

### 1. RAG 기반 LLM SQL 생성 파이프라인 최적화

**문제**: 자연어 → SQL 정확도를 위해 전체 스키마 주입 시 토큰 비용 급증, In-memory Vector로 해결 시 GCP Cloud Run **Cold Start 5초+ 지연** 발생.

**핵심 과제**: 비용 절감 + 지연 시간 보호를 동시에 만족하는 **RAG Retrieval 아키텍처** 구축.

**해결 — 4단계 진화**

| 단계 | 전략 | 해결 | 발생 문제 |
|------|------|------|-----------|
| Phase 1 | 전체 스키마 주입 | — | 토큰 비용 증가 |
| Phase 2 | LLM 자체 판단 | 비용 절감 | 할루시네이션 (엉뚱한 테이블 선택) |
| Phase 3 | In-memory Vector Search | 정확도 + 비용 | Cold Start 지연 (Cloud Run) |
| Phase 4 | DB-native Vector (pgvector) | **모든 문제 해결** | — |

**측정 결과**

| Phase | 응답시간 | 스키마 선택 | 토큰 (대비) | 정확도 |
|-------|---------|------------|------------|--------|
| Phase 1 | 2,628ms | 0ms | 5,847 (기준) | 83% |
| Phase 2 | 3,516ms (+34%) | 1,613ms | 469 (-92%) | 83% |
| Phase 3 | 2,837ms (+8%) | 898ms | 1,130 (-81%) | **100%** |
| Phase 4 | 3,147ms (+20%) | 1,003ms | 923 (**-84%**) | **100%** |

**Cold Start** (Phase 3 → 4 핵심 개선): 임베딩 In-memory 로드 **2,520ms → 0ms (100% 제거)**. RDB와 Vector 인덱스를 같은 PostgreSQL에서 다루는 pgvector의 장점을 극대화한 결정.

---

### 2. Few-shot 도입을 통한 SQL 생성 정확도 개선

**문제**: 예시 없이 스키마만으로 SQL을 생성하니 (1) **재현성 부족** — 자연어 표현이 조금만 달라져도 SQL 형태가 흔들림, (2) **품질 편차** — 동일 질문도 컨텍스트에 따라 결과 변동.

**해결**: 두 메커니즘을 **독립적으로 분리**해 각각 다른 문제를 해결:

- **Few-shot 예시 주입** → 재현성 / 생성 안정성
- **`keywords` 메타 컬럼 (약어·동의어 매핑)** → 표현 변화 강건성

**측정 — 재현성** (동일 질문 5회: "뭉가 2레벨 효과가 어떻게 돼?")

| 항목 | Few-shot 없음 | Few-shot 있음 |
|------|--------------|--------------|
| SQL 구조 일치 | 2/5 | 4/5 (1개 컬럼 누락) |
| 할루시네이션 | 3/5 (없는 테이블 참조) | **0/5** |
| 운영 영향 | 답변 자체 받지 못함 | 정상 응답 |

**측정 — 생성 안정성** (표현 변형 3종: "뭉가 2단계", "레벨2가 뭐야?", "2렙 효과")

| 항목 | Few-shot 없음 | Few-shot 있음 |
|------|--------------|--------------|
| SQL 생성 성공 | 2/3 | **3/3** (동일 패턴) |

---

### 3. 정적/동적 데이터 분리로 스토리지·트래픽 최적화

**문제**: API 응답에 캐릭터 현재 세팅 + 변동 없는 메타데이터(스킬 상세 텍스트, 트라이포드 등)가 **매 수집 주기마다 중복 포함** → 불필요한 트래픽·스토리지 낭비.

**해결**: 정적/동적 분리. 메타데이터는 **1회 일괄 적재(웹 스크래핑 병행) + 패치 시 수동 트리거**로 전환, 동적 데이터만 주기 수집하도록 DAG 분리.

**측정** (1,000명 기준, 단일 캐릭터 82KB → 17KB)

| 항목 | 분리 전 | 분리 후 |
|------|--------|--------|
| 일일 수집 크기 | 약 1.9GB | **약 400MB (79% 절감)** |
| 정적 데이터 빈도 | 매시간 (일 24회) | 패치 시 1회 |
| 운영 부하 | 매 주기 텍스트 파싱 | 트랜잭션만 경량 수집 |

---

### 4. 비정형 API 데이터 정규화 + 쿼리 성능 개선

**문제**: 원본 API의 핵심 정보가 인게임 UI용 HTML 태그(`<FONT>` 등)와 섞인 Nested JSON(`tooltip`)으로 제공 → LLM이 Text-to-SQL 시 비정형 텍스트 파싱 비용으로 **속도 저하 + 할루시네이션 증가**.

**해결**: Airflow Transform 단계에서 **정규식으로 HTML 노이즈 제거 + 핵심 수치 평탄화(Flattening)**, JSONB 텍스트 연산 → **순수 Numeric 연산** 전환, 파생 가능한 총합 테이블 제거로 스키마 직관성 확보.

**측정** (500건 Full Scan, `EXPLAIN ANALYZE`)

| 항목 | 최적화 전 (JSONB) | 최적화 후 (Numeric) |
|------|------------------|--------------------|
| 실행 속도 | 0.644ms | **0.121ms (5.3배 향상)** |
| 메모리 I/O | 36 블록 | 24 블록 |
| 운영 영향 | 토큰 낭비 + 챗봇 지연 | 즉각 실행 + 토큰 최적화 |

## 성과 및 회고

**결과물**: [챗봇 데모](https://gamechatbotfe.vercel.app/)

**배운 점**

- pgvector를 활용한 **RDB + Vector DB 통합 RAG Retrieval** 설계 경험
- **RAG / Text-to-SQL 파이프라인**을 데이터 엔지니어 관점에서 직접 구축 — Retrieval 품질이 정확도·비용을 결정한다는 점 체득
- Few-shot이 단순 프롬프트 기법이 아닌, **운영 안정성(재현성)을 보장하는 메커니즘**임을 측정으로 검증
- **자체 모니터링 도구 개발의 가치** — 외부 도구가 줄 수 없는 도메인 맞춤 인사이트와 빠른 디버깅 사이클

**넥스트 스텝**

- TFT API 확장 안정화 — 멀티 도메인에서 RAG가 게임 컨텍스트를 정확히 라우팅하도록 검증
- BackgroundTasks 한계 도달 시점에 Redis / Celery 점진 마이그레이션
- 데이터 품질 방어 로직 (결측치, 중복, API 타임아웃) 추가 및 문서화
- GitHub Actions 기반 DAG 자동 배포 + 사전 검증 CI/CD 구축
