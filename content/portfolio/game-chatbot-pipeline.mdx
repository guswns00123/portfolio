## 프로젝트 개요

유저들이 게임 정보를 얻기 위해 각기 다른 여러 외부 사이트에 의존해야 하는 **데이터 파편화(Fragmentation) 문제**를 해결하고자 했습니다. 기존의 정적인 표 형태의 정보 제공 방식에서 벗어나, 유저가 자연어로 질문하면 LLM이 필요한 데이터를 즉시 추출 및 가공해 답변하는 **'대화형 데이터 탐색 환경'**을 구축했습니다.

로스트아크 API를 선 적용하여 개발 중이며, 타 게임으로의 확장을 고려한 구조로 설계했습니다.

### 나의 역할

2인 팀 프로젝트 중 **데이터 엔지니어링 및 AI 데이터 아키텍처 영역 100% 전담**

- 외부 게임 API 데이터 수집 및 정제
- Apache Airflow 기반 자동화된 데이터 파이프라인 구축
- PostgreSQL 중심의 RDB 설계 및 데이터 적재
- LLM RAG / Text-to-SQL을 위한 Vector DB 설계 및 임베딩 처리

## 아키텍처

![게임 챗봇 데이터 파이프라인 아키텍처](/images/game-chatbot-architecture.png)

### 데이터 흐름

1. **Ingestion & ETL**: 로스트아크 Open API → Python ETL (Airflow 스케줄링) → PostgreSQL 적재
2. **Service & AI**: 사용자 질의 (Next.js) → FastAPI 백엔드 → pgvector 벡터 검색 및 Text-to-SQL → OpenAI LLM 가공 → 응답 제공

### 핵심 기술 스택 및 도입 배경 (Trade-off)

#### Apache Airflow — 데이터 오케스트레이션

단순 OS 크론탭 대신 도입했습니다. 데이터 종류(캐릭터, 경매장, 거래소)에 따라 수집 주기가 다르고, API Rate Limit으로 인한 실패 시 **재시도(Retry) 로직이 필수적**이었습니다. Airflow의 DAG를 활용해 복잡한 의존성을 관리하고, 파이프라인 실패 시 즉각적인 모니터링이 가능하도록 구성했습니다.

#### PostgreSQL + pgvector — 단일 저장소 전략

LLM 챗봇을 위한 벡터 데이터(임베딩)와 게임의 메타 데이터(관계형 데이터)를 별도의 DB(예: MySQL + Pinecone)로 분리하면 **인프라 관리 리소스와 데이터 정합성 유지 비용이 두 배**로 듭니다. PostgreSQL의 pgvector 익스텐션을 활용하여 RDB와 Vector DB를 단일 저장소로 통합, 시스템 복잡도를 크게 낮추었습니다.

#### FastAPI — AI 백엔드 서버

LLM API 호출과 DB 쿼리(Text-to-SQL)는 대표적인 **I/O Bound 작업**입니다. FastAPI의 비동기(Asynchronous) 처리 지원을 통해 다수의 사용자 챗봇 질의가 동시에 들어와도 블로킹 없이 병렬 처리할 수 있도록 설계했습니다.

## 데이터 수집 설계

### API Rate Limit 분산 전략

로스트아크 Open API는 **API Key 1개당 1분당 100회**라는 엄격한 수집 제한이 있습니다. 이를 해결하기 위해 2개의 API Key를 목적별로 논리적 분리 적용했습니다.

| API Key | 용도 | 수집 대상 |
|---------|------|-----------|
| Key A | 캐릭터 정보 수집 전용 | 유저 프로필, 장비, 스킬 |
| Key B | 경매장/거래소 전용 | 거래 내역, 시세 데이터 |

이를 통해 수집 병목을 해소하고 안정적인 ETL 파이프라인을 구축했습니다.

### 데이터 갱신 주기 차등화 (Freshness)

데이터의 특성과 변동성에 따라 배치 주기를 세분화하여 **리소스 낭비를 방지**했습니다.

| 데이터 유형 | 수집 주기 | 설계 근거 |
|------------|----------|----------|
| 캐릭터 정보 | 1시간 기본 + On-demand | 유저 챗봇 질의 시 최신 정보 즉시 수집 |
| 경매장 (주요 아이템) | **5분 주기** (Near Real-time) | 가격 변동성이 높아 빠른 갱신 필요 |
| 경매장 (일반) | 1시간 주기 | 변동 폭이 적어 과도한 수집 불필요 |
| 거래소 | 1일 1회 ~ 1시간 | 데이터 변동 폭에 따라 혼용 |

### 데이터 볼륨

경매장/거래소 데이터 기준 **일일 약 5,000건**의 트랜잭션 데이터 적재. 캐릭터 데이터는 사용자의 챗봇 활용도에 비례하여 동적으로 축적되는 구조로 설계했습니다.

## 데이터 설계

### 스키마 구조 개요

총 2개 스키마, 5개 그룹으로 구성됩니다.

| 스키마 | 그룹 | 역할 |
|--------|------|------|
| `public` | 챗봇 세션 | 유저 · 세션 · 메시지 관리 |
| `lostark` | 캐릭터 스냅샷 | 캐릭터 정보 수집 이력 (시계열) |
| `lostark` | 정적 게임 데이터 | 스킬 · 각인 · 룬 마스터 데이터 (인벤 크롤링) |
| `lostark` | 시장 데이터 | 경매장 · 거래소 가격 시계열 |
| `lostark` | RAG / Vector | Text-to-SQL Few-shot + 스키마 메타데이터 임베딩 |

### 핵심 설계 결정

#### 1. 스냅샷 설계 — `(character_name, collected_at)` 복합 PK

캐릭터 데이터를 단순 `UPDATE` 방식이 아닌 **수집 시점별 스냅샷으로 누적**합니다. 덮어쓰지 않고 이력을 보존하기 때문에 "어제와 오늘의 장비 세팅 비교" 같은 시계열 질의가 가능합니다.

#### 2. `character_info_tb` 허브 구조

15개 수집 테이블이 모두 `character_name`을 통해 단일 마스터 테이블을 참조합니다. 캐릭터 목록 관리와 수집 이력을 분리하여 **캐릭터 추가/삭제 시 파급 범위를 최소화**했습니다.

#### 3. 가격 조회 크로스 참조

캐릭터 스냅샷 데이터와 시장 데이터는 출처가 다르지만(API vs 시세 수집) 논리적으로 연결됩니다.

| 참조 방향 | 활용 예시 |
|-----------|----------|
| `armory_engravings_tb.name` → `market_items_tb` | "내 캐릭터가 낀 각인 현재 시세는?" |
| `armory_equipment_tb.name` → `auction_items_tb` | "내 장신구랑 동급 매물 경매가는?" |
