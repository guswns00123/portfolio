## 큰 방향

부서 간 데이터 사일로(Data Silo)로 같은 데이터를 서로 다르게 해석하거나 접근조차 못하는 문제를 실무에서 직접 목격한 이후, **비개발 직군도 셀프서비스로 필요한 데이터에 접근할 수 있는 데이터 민주화(Data Democratization) 플랫폼**을 만드는 데이터 엔지니어가 되는 것이 장기 지향점입니다.

## 단기 목표 (1~2년)

### 기술 확장
- **실시간 처리 스택 실무 경험**: Kafka Streams 또는 Spark Structured Streaming을 이용한 실시간 파이프라인 구축
- **대용량 분산 처리 경험**: 테라바이트 단위 데이터를 Spark로 처리하는 실제 업무
- **데이터 플랫폼 관점의 설계**: 개별 DAG 단위가 아닌 메타데이터 관리·카탈로그·권한까지 포함한 플랫폼 단위 설계

### 조직
- 데이터 플랫폼팀 또는 데이터 인프라팀에서 시니어 엔지니어와 함께 일하며 플랫폼 설계 경험 축적
- 개인 프로젝트(게임 챗봇, Text-to-SQL)에서 다룬 LLM 기반 데이터 접근 기술을 실무 제품에 녹여보고 싶음

### 학습 계획
- **Kafka Streams**: 블로그 정리 + 토이 프로젝트로 검증 (현재 진행 중)
- **dbt / Great Expectations**: 데이터 품질·변환 표준 도구 도입
- **Terraform**: IaC로 인프라 관리 학습 시작
- **CI/CD for Data**: GitHub Actions 기반 DAG 자동 배포 + 사전 검증 파이프라인 구축

## 중기 목표 (3~5년)

[수정 필요: 본인 방향에 맞게 조정]

### 후보 시나리오 — 본인 가치관에 맞는 것 선택 또는 조합

#### 시나리오 A. 데이터 플랫폼 엔지니어 트랙
- 한 팀의 데이터 플랫폼 기능(메타데이터, 카탈로그, 권한, 셀프서비스 BI 연동)을 직접 설계·운영
- Airflow 기반 오케스트레이션 외에 Dagster·Prefect 같은 차세대 도구를 실무 도입한 경험
- 데이터 거버넌스·SLA 정의에 참여

#### 시나리오 B. ML/AI 데이터 인프라 트랙
- LLM·ML 모델을 위한 데이터 파이프라인(feature store, vector store, embedding pipeline) 전문화
- 게임 챗봇 프로젝트에서 다룬 RAG·Text-to-SQL 경험을 실무 제품에 녹여 발전
- 모델 서빙 인프라(BentoML, KServe)까지 경험 확장

#### 시나리오 C. 도메인 전문 DE 트랙
- 특정 도메인(커머스, 핀테크, 미디어 등)의 데이터 플랫폼을 깊이 있게 다루는 전문가
- 도메인 비즈니스 임팩트와 직접 연결되는 데이터 작업

예시 방향:
- "**데이터 엔지니어가 1명뿐이라 멘토링 받기 어려운 환경**" — 1년차에게는 시니어로부터의 학습이 핵심이라 함께 성장할 동료가 있는 조직 선호
