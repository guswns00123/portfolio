## 어필 포인트

### 1. 다양한 도메인 실무 경험
1년 동안 4개 프로젝트를 담당 — **공공 데이터(검역본부)**, **부동산 시세(금융 고객사 납품)**, **외부 판매용 경매 데이터**, **공간 데이터(도로 교차로)**. 각기 다른 사용자(공무원·고객사·분석가)와 직접 소통하며 그들의 요구사항에 맞게 데이터 파이프라인을 설계해본 경험.

### 2. 측정 가능한 개선 성과
레거시 산출 프로세스를 ① 검역본부 **12배(1일 → 2시간)**, ② 부동산 **3배(6시간 → 2시간)**. 단순 코드 작성이 아니라 원인 분석 후 구조적 해결을 우선하는 접근.

### 3. 개인 프로젝트로 LLM·실시간 영역 확장 중
실무에서 못 다룬 영역을 개인 프로젝트로 검증 — **게임 챗봇(pgvector RAG + Text-to-SQL, 토큰 84% 절감)**, **TFT 데이터 파이프라인(Lambda 분산 수집)**. 학습한 내용은 블로그·포트폴리오 사이트에 정리.

## 데이터 엔지니어가 된 계기

CUHK 클라우드 수업에서 넷플릭스가 다루는 페타바이트급 데이터를 접하며 "이 많은 데이터를 어떻게 원활하게 처리하는가"라는 의문을 품었습니다. Hadoop·Spark 등 분산 시스템을 공부하며 데이터가 쪼개지고 다시 합쳐지는 프로세스에 매료되었고, 실무에서 검역본부 공무원들이 예측 시스템을 안정적으로 활용하고, 데이터 사이언티스트·개발팀이 파이프라인 구조를 신뢰하며 사용하는 모습을 보며 이 일의 가치를 확신하게 되었습니다.

## 일하는 방식 (요약)

처음에는 "일단 동작하게 만들자"는 접근이었으나, 경매 데이터 파이프라인에서 도메인 특성을 충분히 파악하지 않고 구조를 만들어 수집 오류가 반복되고 문서화 없는 설계로 팀 소통에 어려움을 겪은 이후, **"도메인 이해 → 설계 문서화 → 팀 공유 → 구현"** 순서로 일하는 습관을 갖고자 항상 노력 중에 있습니다.
