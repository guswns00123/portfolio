## 자기소개 관련

### Q. 간단히 자기소개 해주세요
유현준은 홍콩중문대학교 컴퓨터과학과를 졸업한 1년차 데이터 엔지니어다. 빅밸류 데이터 허브 파트에서 검역본부 가축 질병 예측 파이프라인, 부동산 시세 데이터 파이프라인, 경매 데이터 수집 파이프라인 등을 설계·운영했다. 대학에서 Hadoop·Spark 분산 시스템을 공부하며 데이터 엔지니어링에 관심을 갖게 되었고, 실무에서는 비개발 직군(공무원·고객사)이 안정적으로 활용하는 파이프라인을 만드는 일에 가치를 느꼈다. 실시간·대용량 처리 영역으로 기술을 확장하고, 궁극적으로는 데이터 민주화 플랫폼을 만드는 엔지니어가 되는 것이 목표다.

### Q. 왜 데이터 엔지니어인가요?
CUHK 클라우드 수업에서 넷플릭스의 페타바이트급 데이터 처리 사례를 접한 것이 계기였다. Hadoop·Spark를 공부하며 데이터가 쪼개지고 합쳐지는 분산 처리 프로세스에 흥미를 느꼈고, 실무에 들어와서는 검역본부 공무원들이 본인이 만든 파이프라인을 신뢰하며 사용하는 모습, 분석가·개발팀이 설계한 테이블 구조를 이해하고 활용하는 모습을 보며 이 일의 가치를 확신하게 되었다.

### Q. 어느 회사에서 어떤 일을 했나요?
빅밸류(BigValue) 데이터 허브 파트에서 2025년 3월부터 2026년 2월까지 근무. 4개 프로젝트를 담당 — 검역본부 가축 질병 예측 데이터 파이프라인(9개 DAG), 부동산 시세 데이터 파이프라인(6개 DAG, 일 1,300만 건 격자 데이터), 경매 데이터 수집 파이프라인(3개 서버, 일 2,000~3,000건), 도로 교차로 네트워크 구축(1.8만 → 8.4만 개 확장).

## 프로젝트 관련

### Q. 가장 자신 있는 프로젝트는 무엇인가요?
검역본부 가축 질병 예측 데이터 파이프라인. 폐쇄망이라는 제약 환경에서 Oracle → PostgreSQL → Airflow → LLM/예측모델 아키텍처를 설계해 레거시 산출 프로세스를 12배 개선(1일 → 2시간)한 경험이다. 원격 운영이 필수라 테이블별로 task를 분리하고 장애 대응 로그 체계를 설계한 점이 스스로도 잘한 결정이라고 본다.

### Q. 기술적으로 가장 어려웠던 경험은?
경매 데이터 파이프라인 IP 차단 이슈. 단순히 프록시를 추가하는 대신 "수집 대상 최적화"로 접근했다 — 어차피 필요 없는 사건(이미 낙찰 완료 등)까지 크롤링하던 구조를 일 12,000건에서 4,000건 타겟 수집으로 줄여 속도를 2배 향상시켰다. 현상 대응이 아닌 원인 분석을 통한 해결이라 기억에 남는다.

### Q. 실패한 경험이 있나요?
경매 데이터 파이프라인 초기 설계. 도메인 특성(취하·기일변경 등 사건 상태 변경)을 충분히 파악하지 않고 크롤링 구조를 만들어 수집 오류가 반복됐고, 테이블 설계를 문서화 없이 진행해서 팀원 설명에 큰 어려움을 겪었다. 이후 "도메인 이해 → 설계 → 소통 → 구현"의 순서로 일하는 습관을 갖게 되었다.

### Q. 개인 프로젝트 중 설명하고 싶은 것은?
게임 맞춤형 LLM 챗봇 데이터 파이프라인 (진행 중). 로스트아크 API·크롤링 데이터를 pgvector에 임베딩해서 RAG + Text-to-SQL 기반 챗봇을 만들고 있다. 실무에서 다루지 못한 LLM 연동, pgvector 기반 유사도 검색, FastAPI + Next.js 풀스택 구성 등을 경험하는 중이다.

## 기술 관련

### Q. Airflow 경험을 구체적으로?
총 15개 이상 DAG 운영 경험. 테이블별 task 분리, Variable 기반 dev/prod 환경 분리, on_failure_callback으로 Discord 실패 알림 공통화, 재시도 정책·의존성 설계 등. 실무에서 가장 많이 사용한 스택으로, Airflow 운영 자동화와 알림 체계 설계에 자신 있다.

### Q. Spark나 Kafka 실무 경험은?
실무 경험은 없다. Spark는 CUHK 빅데이터 수업에서 MapReduce 기반 알고리즘(K-means, SON, PCY)을 구현했고, 개인 TFT 프로젝트에서 SparkSQL로 집계 작업을 했다. Kafka는 현재 블로그에 개념 정리를 하며 학습 중이다. 실시간·대용량 처리는 다음 커리어 단계에서 실무로 경험하고 싶은 영역이다.

### Q. 파이프라인 설계 시 가장 신경 쓰는 것은?
**실패 가시성과 재시작 용이성**. 테이블 단위로 task를 분리해서 장애가 나도 영향 범위를 좁히고, 로그를 남겨 원격에서도 원인 파악이 가능하게 설계한다. 검역본부 파이프라인이 폐쇄망·원격 운영 환경이었기 때문에 특히 강화된 습관이다.
