### Q. 왜 데이터 엔지니어인가요?
대학 클라우드 수업에서 넷플릭스가 페타바이트급 데이터를 처리하는 사례를 접한 게 계기였습니다. 그때 "이 많은 데이터를 어떻게 안정적으로 다루는 거지?"라는 호기심이 생겼고, Hadoop이나 Spark를 공부하면서 데이터가 쪼개지고 다시 합쳐지는 분산 처리 과정 자체가 정말 흥미로웠습니다. 그러다 실무에 들어와서, 검역본부 공무원분들이 제가 만든 파이프라인을 신뢰하면서 사용하시는 모습이나 분석가·개발팀이 제가 설계한 테이블 구조를 잘 활용하시는 모습을 보면서 이 일의 가치를 더 확신하게 됐습니다.

### Q. 어느 회사에서 어떤 일을 했나요?
2025년 3월부터 2026년 2월까지 빅밸류라는 회사의 데이터 허브 파트에서 근무했습니다. 1년 동안 네 개 프로젝트를 담당했는데, 검역본부 가축 질병 예측 데이터 파이프라인은 9개 DAG로 운영하면서 레거시 산출 프로세스를 12배 개선했고, 부동산 시세 데이터 파이프라인은 6개 DAG로 일 1,300만 건 격자 데이터를 처리하면서 6시간 걸리던 산출을 2시간으로 줄였습니다. 그 외에 경매 데이터 수집 파이프라인을 3개 서버로 일 2,000~3,000건 안정적으로 수집했고, 도로 교차로 네트워크를 1.8만 개에서 8.4만 개까지 4.6배 확장하는 작업도 했습니다.

### B-1. 설계 결정·판단 관련

#### Q. 설계 결정을 내릴 때 가장 중요하게 보는 것은?
저는 **유지보수성과 운영 가시성**을 가장 중요하게 봅니다. 단기적인 구현 속도나 성능보다는 "장애가 났을 때 빠르게 원인을 찾고 복구할 수 있는 환경을 만들 수 있는가"를 우선 생각합니다. 특히 검역본부 파이프라인이 폐쇄망에 원격 운영 환경이었다 보니 이런 관점이 더 강화됐고, 그래서 테이블 단위로 task를 분리하거나 구조화된 로그를 남기고 실패 알림 체계를 잡는 부분에 시간을 많이 투자하는 편입니다.

#### Q. 본인이 내린 설계 결정 중 후회하는 게 있나요?
경매 데이터 수집 파이프라인을 처음 만들 때의 설계가 좀 후회됩니다. 처음에는 과거에 사용하던 테이블 구조를 그대로 따라가려고 했는데, 알고 보니 그 구조가 도메인에 대한 충분한 이해 없이 만들어진 거라 테이블 간 연결이 어렵고 논리적으로도 이해하기 힘든 구조였습니다. 그래서 수집을 진행하면서 에러가 나거나 분석가의 요청사항이 들어올 때마다 테이블 구조를 계속 바꿔야 했던 기억이 있습니다. 그때 만약 구조 설계 전에 도메인을 충분히 이해하고 시작했더라면 처음부터 제대로 설계할 수 있었을 텐데, 라는 아쉬움이 많이 남았고, 그 이후로는 무조건 도메인 학습부터 하고 설계로 넘어가는 습관을 가지게 됐습니다.

#### Q. 시간이 더 있었다면 게임 챗봇에 무엇을 추가했을 건가요?
지금 챗봇이 답변할 수 있는 수준은 기본적인 정보 조회나 간단한 기초 통계 분석 정도라서, 좀 더 고수준 분석까지 답할 수 있게 만들고 싶었습니다. 그러려면 각 게임에 대한 구체적인 도메인 지식을 챗봇이 학습할 수 있는 구조로 만들어야 한다고 보고 있고, 지금도 지속적으로 보강하려고 노력하고 있습니다. 그리고 현재는 두 게임 정도만 지원하는데, 앞으로 게임을 계속 추가해서 사용자가 즐기는 게임에 대한 지식을 이 사이트 하나에서 다 얻을 수 있는 구조로 확장하고 싶습니다.

### B-2. 운영·신뢰성 관련

#### Q. 가장 큰 장애 경험은? 어떻게 대응하셨나요?
검역본부 프로젝트 진행 중에 김천에 출장을 내려가서 직접 데이터 파이프라인을 구축한 적이 있는데, 그때 시간이 굉장히 빠듯한 상황에서 geometry 데이터를 미리 DB에 만들어둬야 하는 작업이 있었습니다. 그런데 전국 단위 geometry 연산이라 한 번에 처리하면 1일 넘게 걸릴 것 같았고, 실제로 한 번에 돌리니까 DB가 계속 뻑이 가는 상황이었습니다. 그래서 우선 geometry 인덱스를 걸어서 연산을 단순화하고, 전국을 한 번에 처리하는 게 아니라 시도 단위로 쪼개서 순차 처리하는 방식으로 전환했습니다. 그 결과 1일 걸릴 작업을 6시간 내외로 단축할 수 있었고, 시간 안에 출장 일정을 마무리할 수 있었습니다.

#### Q. 본인 파이프라인의 가장 약한 부분은?
솔직히 말씀드리면 **테스트 커버리지** 부분이 가장 약하다고 생각합니다. 실무에서는 데이터 정합성을 검수 task로 자동 검증하는 정도까지는 했지만, Python 함수 단위의 unit test나 task 간 integration test 같은 표준적인 테스트 체계는 부족했습니다. 어떤 종류의 테스트를 어떤 기준으로 구축해야 하는지에 대한 지식이 아직 부족한 상태라, 다음 단계에서 꼭 공부하고 실제로 적용해보고 싶은 영역입니다.

#### Q. 운영 중인 파이프라인의 비용(클라우드, 컴퓨팅)을 의식한 적 있나요?
개인 프로젝트인 게임 챗봇에서 비용을 정말 많이 신경 썼습니다. AI 모델 API 비용이랑 Airflow를 돌리는 GCP 컴퓨터 서버 비용 두 가지가 주요 지출이었는데, 특히 AI 모델은 좋은 모델을 쓰면 답변 품질은 좋지만 비용이 많이 나가고, 낮은 모델을 쓰면 가격은 훨씬 이득이지만 성능이 떨어지는 trade-off가 명확했습니다. 그래서 **최대한 낮은 등급 모델로도 높은 품질의 답변을 낼 수 있도록 프롬프트와 데이터 주입 구조를 명확하게 설계**하는 데 집중했고, 이 과정이 챗봇 프로젝트의 4단계 진화에서 토큰을 84%까지 절감한 핵심 동기였습니다. 실무에서는 비용을 직접 의식할 기회가 많지 않았는데, 시니어로 갈수록 중요한 역량이라고 인식하고 있습니다.

### B-3. 안 해본 영역 (솔직 어필 + 학습용 개념 정리)

> 직접 경험이 없는 영역은 솔직하게 인정하되, **개념과 학습 방향**을 함께 제시하는 답변 패턴.

#### Q. CI/CD 경험은?
**CI/CD라는 건** Continuous Integration / Continuous Delivery의 약자로, 코드 변경이 생기면 자동으로 테스트하고 배포까지 이어지는 파이프라인을 의미합니다. 데이터 엔지니어 입장에서 보면 ① **DAG 코드 변경 시 자동으로 문법·DAG 파싱 검증**, ② **테스트 통과 시 운영 서버에 자동 배포**, ③ **배포 후 헬스체크와 롤백** 같은 흐름이 핵심입니다.

솔직히 실무에서는 단순한 git pull 기반 배포 수준이었고, 본격적인 CI/CD 파이프라인은 개인 프로젝트에서만 다뤄봤습니다. 포트폴리오 사이트는 Vercel의 자동 배포 기능을 이용했고, TFT 프로젝트는 GitHub Actions로 GCP VM에 SSH 접속해서 git pull 하는 정도까지 구성했습니다. 다음 학습 목표는 **Airflow DAG 자동 배포 + 사전 검증(파싱·DAG validation 테스트)** 파이프라인을 구축해보는 것입니다.

#### Q. Terraform / IaC 경험은?
**IaC는** Infrastructure as Code의 약자로, 서버나 네트워크, DB 같은 인프라를 클릭이 아니라 **코드로 정의해서 버전 관리하고 자동 배포**하는 방식입니다. Terraform이 대표적인 도구이고, 같은 인프라를 dev/staging/prod 환경에 동일하게 재현하거나, 인프라 변경 이력을 git으로 추적할 수 있다는 게 핵심 가치입니다.

저는 실무와 개인 프로젝트 모두 수동으로 GCP 콘솔에서 인프라를 구성하는 단계라, Terraform 같은 IaC 경험은 아직 없습니다. 다만 데이터 플랫폼 팀에서 일하게 되면 가장 먼저 익혀야 할 영역이라고 보고 있어서, 우선 Terraform 공식 튜토리얼로 GCP 리소스 하나를 코드로 정의해보는 단계부터 시작하려고 합니다.

#### Q. 데이터 거버넌스(PII, GDPR) 경험은?
**데이터 거버넌스**는 한마디로 **"누가, 어떤 데이터에, 어떤 권한으로 접근하고, 그 사용이 어떻게 추적되는가"**를 관리하는 체계입니다. PII(개인 식별 정보)나 GDPR 같은 법규 준수 요건이 있는 데이터는 ① **접근 권한을 역할 기반으로 제한**, ② **누가 언제 조회했는지 감사 로그 기록**, ③ **삭제 요청 시 관련 데이터 일괄 삭제(Right to be forgotten)** 같은 기능이 필수입니다.

직접적인 PII 처리 경험은 부동산 데이터에 일부 개인 식별 정보가 포함돼 있어서 고객사 납품 시 마스킹·집계 처리한 정도로 제한적입니다. 본격적인 거버넌스 체계(접근 제어, 감사 로그, 삭제 요청 처리)를 운영해본 적은 없고, 도메인이 명확해지는 단계에서 학습이 필요한 영역으로 인식하고 있습니다.

#### Q. dbt / Great Expectations / Marquez 같은 도구 경험은?
세 도구 모두 직접 도입해본 경험은 없지만, 각각이 데이터 엔지니어링에서 어떤 역할을 하는지는 정리하고 있습니다.

- **dbt**는 SQL 기반의 데이터 변환 도구로, "원천 테이블 → 가공 테이블"의 변환 로직을 모듈화하고 의존성 그래프로 관리해주는 도구입니다. 변환 로직을 SQL 모델로 정의하면 dbt가 알아서 순서대로 실행해줍니다.
- **Great Expectations**는 데이터 품질 검증 프레임워크로, "이 컬럼은 null이면 안 된다", "이 값은 0~100 사이여야 한다" 같은 기대값(expectations)을 코드로 선언하고 자동으로 검증해주는 도구입니다.
- **Marquez**는 데이터 lineage(계보) 추적 도구로, 어떤 데이터가 어디서 와서 어떤 변환을 거쳐 어디로 가는지를 자동으로 시각화해줍니다.

저는 데이터 품질 검증은 자체 Python 검수 task로 처리했고, lineage는 Airflow task 의존성과 코드 자체로 사실상 관리한 상태였습니다. 표준 도구로 옮기면 ① 검증 로직이 더 선언적이고, ② lineage가 자동 추적되고, ③ 팀원이 해당 도구의 표준에 익숙하면 학습 비용이 줄어든다는 이점이 있어서, 다음 학습 우선순위 중 하나로 보고 있습니다.

#### Q. 실시간(Streaming) 처리 경험은?
실무 경험은 없습니다. 다만 대학 과제로 **Spark Structured Streaming을 활용해 Kafka로부터 실시간 트위터 데이터를 받아서 5분 윈도우 단위로 인기 해시태그 상위 30개를 집계**하는 프로젝트를 진행해본 경험은 있습니다. 이전에는 RDD 기반 Spark Streaming으로 비슷한 작업을 했었는데, Structured Streaming으로 다시 구현해보면서 윈도우 연산이나 상태 관리 같은 고급 기능을 훨씬 직관적인 API로 처리할 수 있다는 걸 체감할 수 있었습니다. 다만 이건 학습 단계 경험이고 실서비스에서 실시간 파이프라인을 운영해본 적은 없어서, 다음 커리어 단계에서 꼭 다뤄보고 싶은 핵심 영역입니다.

## 섹션 C. 이직 / 지원 / 가치관

### Q. 왜 이직을 고려하나요?
이전 직장에서 다뤘던 데이터의 규모가, 제가 공부했던 분산 처리나 실시간 처리 같은 기술을 실제로 적용해보기에는 좀 작았던 것이 가장 큰 이유입니다. 한 달에 1TB 정도가 쌓이는 일 배치 중심 환경이었는데, 그러다 보니 데이터 엔지니어로서 다음 단계의 기술 확장에 한계가 느껴졌습니다. 더 큰 규모의 데이터를 다뤄보고 싶고, 그러기 위해서는 실시간 처리 영역까지 경험이 확장되어야 한다고 판단해서 이직을 결정하게 됐습니다.

### Q. 어떤 회사를 찾고 있나요?
크게 세 가지를 보고 있습니다. 첫 번째는 **데이터 플랫폼이나 실시간 처리 스택**을 실제로 다루는 팀이고, 두 번째는 **데이터 자체가 의사결정의 핵심**이 되는, 데이터에 힘이 실려 있는 회사입니다. 세 번째는 **함께 공부하고 배워나갈 수 있는 팀 분위기**인데, 1년차 입장에서는 시니어 분들에게 배울 수 있는 환경이 정말 중요하다고 생각하기 때문입니다.

### Q. 일과 삶의 균형 어떻게 생각하시나요?
물론 균형도 중요하다고 생각하지만, 저는 아직 성장 단계에 있는 시점이라 일과 일을 위한 학습에 의식적으로 시간을 투자하고 싶은 마음이 더 큽니다. 사실 1년 동안 일했던 시기와 잠시 백수로 지냈던 시기를 둘 다 경험해보니까, 개인적으로는 일하는 시기가 훨씬 더 재밌고 사는 맛이 있다고 느꼈습니다. 다만 오버타임이 일상화되는 문화는 지속 가능하지 않다고 보고, 의식적인 투자와 일상화된 야근은 다른 문제라고 생각합니다.

### Q. 스트레스 해소 방법은?
업무에 집중이 잘 안 될 때는 보통 노래를 들으면서 일을 하면서 효율을 끌어올리는 편이고, 스트레스를 좀 받았다 싶을 때는 친구들과 잡담을 하거나 게임을 하면서 푸는 편입니다. 적당히 환기하고 다시 일에 집중하는 사이클이 저한테는 잘 맞는 것 같습니다.

### Q. 5년 후 어떤 모습이 되고 싶나요?
저는 데이터 엔지니어로서 같은 팀원뿐만 아니라 다른 파트의 팀원들이 데이터 관련해서 무언가 궁금한 게 생겼을 때 가장 먼저 찾아오는, 그런 사람이 되고 싶습니다. "이 사람이 회사에 없으면 어쩌지"라는 걱정을 하게 만드는 사람이 되고 싶다는 게 솔직한 목표인데, 사실 전 직장 팀장님이 그런 영향력을 가진 분이셨고, 제가 그분에게서 받은 좋은 영향이 컸기 때문에 저도 후배나 상사 분들께 비슷한 좋은 영향을 줄 수 있는 사람이 되고 싶습니다. 기술적으로 말씀드리면, **부서 간 데이터 사일로(Data Silo)가 없는 회사**를 만들어보는 것이 5년 후 도달하고 싶은 모습입니다.

### Q. 마지막 한마디 / 면접 마무리
오늘 이렇게 좋은 자리를 마련해 주셔서 정말 감사합니다. 면접관님들 덕분에 제 부족한 점들과 동시에 제가 왜 이 직무에 적합한지 다시 한번 말씀드릴 수 있어서 뜻깊은 시간이었습니다. 입사하게 된다면 빠르게 적응해서 팀에 기여하고 싶고, 동시에 시니어 분들의 지식과 경험을 적극적으로 흡수해서 회사에 더 큰 기여를 할 수 있는 사람이 되도록 노력하겠습니다. 감사합니다.
