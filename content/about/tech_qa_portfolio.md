## A. 게임 챗봇 데이터 파이프라인 (lostark)

### Q1. pgvector를 선택하셨는데, Pinecone이나 Qdrant 대비 trade-off를 어떻게 평가하시나요?
가장 큰 이유는 **인프라·정합성 비용**이었습니다. 게임 메타데이터(RDB)와 임베딩(Vector)을 별도 DB로 두면 운영할 시스템이 두 개로 늘고, 두 DB 사이 데이터 동기화 문제가 생깁니다. pgvector는 단일 DB에서 RDB JOIN과 벡터 검색을 모두 할 수 있어 챗봇처럼 **메타데이터와 벡터를 함께 쿼리하는 패턴**에 적합합니다. 단, **수억 row 규모로 커지면** Pinecone 같은 전용 서비스의 처리량·관리 편의성이 더 유리해질 거라 보고, 임계점을 모니터링하는 게 다음 과제입니다.

### Q2. Phase 4 (DB-native Vector)의 응답 시간 3,147ms도 짧은 편은 아닌데, 더 줄일 여지가 있나요?
네, 분석해보면 ① **LLM 호출 자체**가 1.5~2초로 가장 큰 비중이고, ② 나머지가 DB 쿼리·임베딩 변환입니다. 줄일 수 있는 방향은 ① **gpt-4o-mini 같은 경량 모델로 일부 단계 대체**, ② **자주 쓰이는 질의 패턴 캐싱**(같은 질문이 반복될 가능성), ③ **스트리밍 응답으로 체감 latency 개선**. [수정 필요: 실제 검토했거나 적용 중인 항목]

### Q3. Text-to-SQL 정확도 100%라고 하셨는데, 측정 데이터셋의 규모와 구성은 어떻게 되나요?
[수정 필요: 측정 데이터셋 — 몇 개의 테스트 쿼리, 어떻게 구성, 어떤 기준으로 정답 판정] 솔직히 말하면 평가 데이터셋 규모가 크지 않아 **"제한된 도메인 내에서의 100%"**라는 한계가 있습니다. 실서비스에 들어가면 사용자가 던지는 다양한 변형 질의에서 정확도가 떨어질 가능성이 있고, 이를 위해 **로그 기반 실패 케이스 수집 → 평가 데이터셋 확장 → 모델·프롬프트 개선** 사이클이 필요하다고 보고 있습니다.

### Q4. API Key 2개 분리 전략, Key가 더 필요해지면 어떻게 확장하시겠어요?
지금은 "캐릭터" / "경매·거래소" 두 도메인으로 논리 분리한 구조라, 도메인이 늘면 자연스럽게 Key를 추가할 수 있습니다. 다만 ① **수평 확장(같은 도메인에 Key 여러 개)**이 필요해지면 라운드로빈 또는 사용량 기반 분배 로직이 필요하고, ② **Key별 사용량·실패율 모니터링**도 같이 해야 합니다. 현재 구조는 N=2까지 단순하게 잘 동작하지만, N이 늘면 별도 **Key Pool 관리 모듈**로 추출하는 게 맞다고 봅니다.

### Q5. `(character_name, collected_at)` 복합 PK 설계의 trade-off는?
**장점**: 시점별 스냅샷이 쌓여서 시계열 비교(어제 장비 vs 오늘 장비) 같은 질의가 가능, 멱등성 자연스럽게 확보. **단점**: 데이터가 누적되어 **스토리지가 시간 비례로 증가**, 최신 정보만 필요한 쿼리도 created_at 필터링이 필요해 인덱스 의존도가 높아짐. 저는 게임 도메인 특성상 "변화 추적"이 핵심 가치라 누적 비용을 감수했지만, 단순 lookup만 필요한 도메인이라면 UPSERT가 더 적절합니다.

### Q6. 정적/동적 데이터 분리 후 재결합이 필요한 쿼리는 어떻게 처리하나요?
분리는 **수집 파이프라인 레벨**에서만 이뤄지고, 쿼리 레벨에서는 정상적으로 JOIN으로 재결합합니다. 예: "내 캐릭터의 각인 시세"는 `armory_engravings_tb`(동적) ⨝ `engraving_master_tb`(정적) ⨝ `market_items_tb`(시장)로 묶어 조회. 정적 테이블은 자주 안 바뀌므로 **PostgreSQL의 buffer cache hit**이 잘 되어 JOIN 비용이 거의 없었습니다(EXPLAIN ANALYZE 0.121ms 수준).

### Q7. 데이터 품질 방어 로직을 안 넣었다고 하셨는데, 운영에서 가장 먼저 망가질 시나리오는?
가장 현실적인 시나리오는 **로스트아크 API 응답 스키마 변경**입니다. 게임 패치로 새 필드가 추가되거나 기존 필드 타입이 바뀌면 파싱 단계에서 터지거나 잘못된 데이터가 적재됩니다. 다음으로는 ① **API 타임아웃 시 부분 적재**, ② **임베딩 생성 실패 시 RAG 결과 누락**. 최우선 보강은 **수집 단계 schema validation + 적재 후 row 수·null 비율 검수 task**라고 보고 있습니다.

### Q8. FastAPI를 선택한 이유와, Django REST나 Flask 대비 적합성은?
LLM 호출과 DB 쿼리는 **I/O Bound**라 비동기 처리가 큰 이점이 있습니다. FastAPI는 ASGI 기반 native async 지원이라 Flask(WSGI)보다 동시 처리 효율이 좋고, Django REST보다 가볍고 학습 곡선이 낮습니다. 또한 **Pydantic 기반 type validation**이 LLM 응답 파싱·검증에도 잘 맞아서 선택했습니다.

### Q9. Airflow와 FastAPI 사이 장애 격리는 어떻게 되어있나요?
두 시스템은 **DB만 공유**하고 직접 통신은 없습니다. Airflow가 죽어도 FastAPI는 기존 적재된 데이터로 계속 응답할 수 있고(다만 신선도가 떨어짐), FastAPI가 죽어도 Airflow는 정상적으로 데이터 적재를 계속합니다. **DB가 단일 장애점**이라는 점은 단점이고, 실서비스 단계에서는 read replica 분리를 고려할 수 있습니다.

### Q10. Cold Start 0ms라고 하셨는데, DB 자체 쿼리 latency도 분석하셨나요?
네, **Phase 3에서 Cold Start 2,520ms를 0ms로** 줄인 게 핵심이었지만, 그 자리를 차지하는 **DB 벡터 검색 자체의 latency**가 새로 측정 대상이 됩니다. ivfflat/hnsw 인덱스가 적용된 상태에서 단일 검색은 수 ms 수준으로 들어왔습니다. 다만 데이터가 늘어나면 인덱스 빌드·재구축 비용이 운영 부담으로 등장할 거라 보고 있습니다. [수정 필요: 실제 측정값]

## B. TFT 챌린저 데이터 파이프라인

### Q11. Lambda 동시 호출을 하필 3개로 결정한 근거는?
[수정 필요: 실제 결정 근거] 솔직히 정량적 분석보다는 **시행착오로 안전한 값**을 찾은 결과입니다. Riot API의 분당 호출 제한 안에서 동시성을 높일수록 빨리 끝나지만, 너무 높이면 **API 응답이 burst로 몰려 일시적으로 limit 초과**가 발생했습니다. 3개가 안정적으로 limit을 지키면서 처리 시간도 허용 범위였던 sweet spot이었습니다. 더 정밀하게 한다면 **token bucket** 같은 rate limiter를 적용하는 게 맞습니다.

### Q12. Lambda → PostgreSQL 직접 연결 시 connection pool 문제는 없었나요?
[수정 필요: 실제 어떻게 처리했는지] Lambda는 **요청마다 새 컨테이너가 떠서 connection을 새로 여는** 패턴이라 connection 폭증 위험이 있습니다. 당시 동시성을 3으로 제한한 것이 실질적으로 connection 수도 제한해주는 효과가 있어 큰 문제가 없었지만, 본격적인 production이라면 **RDS Proxy 또는 PgBouncer**를 앞단에 두는 게 표준입니다.

### Q13. 6개 테이블 정규화 — JOIN 비용과 비정규화 trade-off는 어떻게 보셨나요?
정규화 우선으로 가져간 이유는 ① **유닛/증강체/특성이 매 게임마다 반복**되어 비정규화 시 데이터 폭증, ② **분석 쿼리가 정해진 패턴**이라 자주 쓰는 JOIN은 인덱스로 충분히 빠름. 다만 **대시보드처럼 자주 조회되는 집계는 SparkSQL로 일배치 처리해서 결과 테이블에 적재**하는 식으로 분리해, 정규화의 단점을 피했습니다.

### Q14. 증분 수집(Incremental Load) 미구현이라셨는데, 지금 구조에서 어떻게 추가할 수 있을까요?
유저 데이터는 **마지막 수집 시점 이후 LP/티어 변경된 유저**만 갱신하는 방식이 가능합니다. 경기 데이터는 매 유저마다 **마지막으로 수집한 game_id 이후 새 경기**만 가져오면 됩니다. 구조적으로는 ① 메타 테이블에 `last_collected_at` / `last_game_id` 저장, ② Lambda에서 이 값을 읽어 API 호출 범위를 제한, ③ 성공 시 메타 갱신. 큰 변경 없이 추가 가능한 영역입니다.

### Q15. SparkSQL 대신 PostgreSQL 집계 쿼리로 처리해도 됐을 것 같은데 왜 Spark였나요?
[수정 필요: 솔직한 답변 — 학습 목적이었는지, 데이터 규모상 필요했는지] 솔직히 말하면 데이터 규모는 PostgreSQL로도 충분했지만, **분산 처리 학습 경험을 쌓고 싶어서** SparkSQL을 도입했습니다. 실제 production에서는 데이터 규모와 운영 복잡도를 따져 적절한 스택을 선택해야 한다는 점을 이 프로젝트에서 배웠습니다.

## C. 빅밸류 실무 프로젝트

### Q16. 검역본부 12x 개선의 진짜 원인이 알고리즘인지, 병렬화인지, 인프라인지?
가장 큰 비중은 **알고리즘과 SQL 튜닝**이었습니다. 레거시는 농장별 위험도 산출 시 행 단위 반복 쿼리(loop in Python)였는데, 이를 **set-based SQL 연산으로 재작성**하고 중간 결과를 임시 테이블로 캐싱했습니다. 추가로 Airflow에서 농장 그룹별 task로 병렬화한 것이 보조적으로 기여했고, 인프라(서버 사양) 변경은 없었습니다. [수정 필요: 실제 비중 — 예 "쿼리 70%, 병렬화 30%"]

### Q17. 9개 DAG 사이의 의존성과 SLA는 어떻게 관리하셨나요?
DAG 간 의존성은 **`ExternalTaskSensor`** 또는 **trigger DAG** 패턴으로 관리했습니다. 원천 수집 DAG가 끝나야 위험도 산출 DAG가 시작하고, 위험도 산출이 끝나야 View 제공 DAG가 시작하는 직렬 의존 구조입니다. SLA는 [수정 필요: 명시적으로 `sla` 파라미터를 썼는지, 아니면 알림으로만 관리했는지] 명시적인 SLA 파라미터보다는 **각 DAG의 평균 실행 시간 + 알림 채널**로 운영했습니다.

### Q18. 폐쇄망 환경에서 Airflow 운영의 실제 가장 큰 어려움은?
**패키지 설치와 외부 도구 활용 제약**이 가장 컸습니다. pip install이 안 되는 환경이라 ① 외부에서 wheel 파일을 만들어 와서 오프라인 설치, ② Docker 이미지를 미리 외부에서 빌드해서 가져오는 방식. 또한 **외부 API·SaaS 모니터링 도구**(Datadog 등)를 못 써서 자체 로그 수집·Discord webhook 기반 모니터링을 자체 구축해야 했습니다.

### Q19. 부동산 1,300만 건 격자 데이터 — 파티셔닝 적용했나요? 안 했다면 왜?
[수정 필요: 실제 적용 여부] 솔직히 파티셔닝은 적용하지 않았습니다. 이유는 ① **월 1회 산출 + 전체 격자를 한 번에 처리**하는 패턴이라 partition pruning의 이점이 적었고, ② **인덱스(B-tree on PNU + 격자ID)와 적절한 SQL 작성**으로 6시간 → 2시간 개선이 가능했기 때문입니다. 만약 일 단위 갱신·조회 패턴이 됐다면 시간 기반 파티셔닝이 필요했을 것입니다.

### Q20. 6시간 → 2시간은 어떻게 측정하셨나요? 입력 데이터는 동일했나요?
**같은 월의 동일 입력 데이터**를 기준으로 ① 레거시 코드 실행 시간, ② 리팩토링 후 실행 시간을 각각 5회 측정해 평균을 비교했습니다. 변화 요인은 ① **불필요한 중간 테이블 제거**, ② **반복 SUBQUERY를 CTE로 일원화**, ③ **JOIN 키에 명시적 인덱스 추가**, ④ **PNU 매칭 로직을 Python에서 SQL로 이동**. 인프라·데이터는 동일하게 유지해 변화 원인이 코드 개선이라는 것을 확인했습니다.

### Q21. 경매 IP 차단 대응에서 클라우드 이전 외에 robots.txt나 ToS 검토는 하셨나요?
[수정 필요: 솔직한 답변] 솔직히 robots.txt는 확인했지만 **ToS의 법적 해석까지는 깊이 검토하지 못했습니다**. 외부 판매용 데이터라는 비즈니스 결정이 회사 차원에서 이뤄졌고, 저는 기술적 수집·정제 영역만 담당했습니다. 다만 **수집 부담을 최소화**하는 방향(일 12,000건 → 4,000건 타겟 수집)으로 가는 것이 단순한 차단 우회보다 윤리적·기술적으로 옳다고 봤습니다.

### Q22. 자동 검수 체계의 false positive 비율은?
[수정 필요: 실제 비율 또는 추정] 정확한 측정은 못 했지만, 초기에는 **임계값이 너무 타이트해 정상 데이터도 결함으로 잡히는 케이스**가 일부 있었습니다. 운영하면서 ① **결함 유형별 임계값 조정**, ② **알림 등급 분리(Critical / Warning)**로 false positive로 인한 알림 피로를 줄였습니다.

### Q23. 비정규화 테이블 설계 시 데이터 정합성은 어떻게 보장하셨나요?
경매 테이블은 **수집 단계가 단일 source(웹 크롤링)**라서 정합성 이슈가 비교적 단순했습니다. 같은 사건의 정보가 여러 곳에서 들어와 충돌하는 구조가 아니었기 때문입니다. 대신 ① **복합 PK로 중복 차단**, ② **수집 시점에 source URL·타임스탬프 함께 저장**해 추후 검증 가능하게 했습니다. 다중 source 환경이라면 비정규화는 더 신중해야 한다고 봅니다.

## D. 도로 교차로 네트워크

### Q24. 1.8만 → 8.4만 개로 4.6배 확장 — 정확도(false intersection) 검증은 어떻게?
[수정 필요: 실제 검증 방법] QGIS 시각화로 **샘플 지역의 실제 도로망과 교차하는지 육안 검증**한 후, 분석팀과 함께 사용 케이스에 적합한지 검토했습니다. 정량적 정확도 메트릭은 산출하지 않았고, 이 부분은 회고하면 보강이 필요한 영역입니다.

### Q25. PostGIS의 SRID 변환을 자주 하셨다는데, 어떤 이슈가 있었나요?
한국 좌표계는 **EPSG:5179 (UTM-K)**가 표준인데, 외부 데이터는 **EPSG:4326 (WGS84)**으로 들어오는 경우가 많아 변환이 필요합니다. 거리 계산을 WGS84(경위도)로 하면 km 단위 오차가 생기기 때문에, **거리 연산 전에 반드시 미터 단위 좌표계로 변환**하는 패턴을 만들었습니다.

## E. 포트폴리오 사이트 자체 (Next.js + 챗봇)

### Q26. 포트폴리오 사이트 챗봇은 어떤 구조인가요?
- **Frontend**: Next.js 16 App Router, SSE 스트리밍 UI
- **Backend**: API Route(`/api/chat`) — Google Generative AI SDK
- **모델**: 처음에는 Ollama 로컬 Gemma → 배포 환경 호환을 위해 **Google AI Studio의 Gemma 3 27B**로 전환
- **컨텍스트 주입**: `content/about/*.md`, `content/portfolio/*.mdx`를 빌드 시점에 읽어 system prompt에 포함
- **보안**: IP 기반 in-memory rate limit (1분당 5회)

### Q27. Vercel 서버리스 환경에서 in-memory rate limit의 한계는?
서버리스는 **인스턴스가 cold start로 새로 뜨면 메모리가 초기화**되고, 동시에 여러 인스턴스가 떠 있으면 각자 별개의 카운터를 가집니다. 즉 실제 차단 효과가 의도보다 약합니다. 트래픽이 늘어나면 **Upstash Ratelimit (Redis 기반)**으로 전환할 계획이지만, 현재는 사용량이 매우 낮아 in-memory로도 충분하다고 판단했습니다.

### Q28. Gemma가 systemInstruction을 지원하지 않는 한계는 어떻게 우회했나요?
첫 user 턴에 **시스템 프롬프트를 user 메시지로 위장해서 주입**하고, 그 다음 model 턴에 **"네, 준비되었습니다."라는 가짜 ack**를 미리 넣어두는 패턴으로 우회했습니다. 이후 진짜 사용자 메시지가 시작되도록 history를 구성해서 Gemma가 시스템 프롬프트를 학습한 것처럼 동작하게 만들었습니다.

### Q29. 챗봇 컨텍스트(about + portfolio + blog)가 LLM context window를 넘어선 경험이 있나요?
네, 초기에 blog 디렉토리 전체(78KB)를 포함시키니 **about/portfolio 정보가 "lost in the middle" 영역으로 밀려 챗봇이 본인 정보를 못 찾는** 현상이 발생했습니다. 해결로 ① **blog는 옵션으로 분리**(기본 제외), ② **`num_ctx: 12288`로 컨텍스트 윈도우 명시 확장**(Ollama 시절). 장기적으로는 **about도 RAG로 분리해서 질의에 맞는 부분만 주입**하는 게 정공법이라 보고 있습니다.

## F. 횡단 / 일반 시니어 질문

### Q30. 본인 코드의 테스트 전략은? unit test, integration test 비중은?
[수정 필요: 솔직한 답변 — 안 했다면 안 했다고] 솔직히 말하면 실무에서는 **테스트 커버리지가 낮은 편**이었습니다. 검수 task로 데이터 정합성은 자동 검증했지만, Python 함수 단위의 unit test는 핵심 로직(검수, 격자 매칭) 위주로만 작성했고 파이프라인 task의 integration test는 거의 없었습니다. 다음 단계에서는 **dbt test, pytest 기반 task 단위 테스트, Airflow `dag_validation_test`**를 도입하는 게 우선순위라고 보고 있습니다.

### Q31. CI/CD 적용 경험은?
[수정 필요: GitHub Actions 적용 여부] 실무에서는 단순한 git pull 기반 배포에 가까웠고, 본격적인 CI/CD 파이프라인은 개인 프로젝트 수준에서 학습 중입니다. 포트폴리오 사이트는 Vercel의 자동 배포를, TFT 프로젝트는 GitHub Actions로 GCP VM SSH 배포를 구성한 정도입니다. **DAG 자동 배포 + 사전 검증(파싱·DAG validation)**을 다음 학습 목표로 삼고 있습니다.

### Q32. Terraform / IaC 경험은?
실무·개인 프로젝트 모두 **수동 인프라 구성** 단계입니다. Terraform은 학습 시작 단계로, 데이터 플랫폼팀에서 일하게 되면 가장 먼저 익혀야 할 영역이라 보고 있습니다. 현재는 Docker Compose로 로컬·서버 환경을 코드화하는 수준입니다.

### Q33. 데이터 거버넌스(PII, GDPR 등) 관련 경험은?
직접적인 PII 처리 경험은 제한적입니다. 부동산 데이터에 일부 개인 식별 정보가 있을 수 있어 **고객사 납품 시 마스킹·집계 처리**한 정도입니다. GDPR·국내 개인정보보호법에 맞춘 본격적인 거버넌스(접근 제어, 감사 로그, 삭제 요청 처리 등) 경험은 없고, 다음 단계에서 학습이 필요한 영역입니다.

### Q34. 운영 중인 파이프라인의 비용(클라우드, 컴퓨팅)을 의식한 적 있나요?
TFT 프로젝트에서 **Lambda 호출 횟수와 S3 저장 비용**을 무료 티어 안에서 운영하기 위해 ① 호출 분산, ② 압축 적재, ③ 오래된 데이터 lifecycle policy 적용 같은 결정을 했습니다. 게임 챗봇에서는 **OpenAI API 토큰 비용이 가장 큰 변수**였고, 4단계 진화 과정의 핵심 동기 자체가 토큰 비용 최적화(84% 절감)였습니다. 실무에서는 비용 의식을 의식적으로 가져갈 기회가 많지 않았는데, 이 부분도 시니어로 갈수록 중요한 역량이라 인식하고 있습니다.
