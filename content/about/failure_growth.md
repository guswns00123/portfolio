# 실패 / 갈등 → 성장 경험

> "실패한 경험이 있나요?" / "협업 중 갈등 경험은?" 질문 대비.
> 핵심: **문제 인식 → 원인 분석 → 개선 행동 → 성장**의 흐름이 보여야 함.
> 단순한 안 좋은 일 나열이 아니라 **이후 어떻게 달라졌는지**가 중요.

---

## 사례 1. 기술적 실패와 극복: '빠른 구현'에서 '안정적인 설계'로의 전환

### Situation (상황)
경매 데이터 수집 파이프라인 구축 프로젝트를 맡았을 때, 도메인에 대한 깊은 이해보다는 '빠르게 동작하는 코드를 완성하는 것'을 최우선 목표로 삼았습니다.

### Task (문제)
일단 구현부터 마친 후 운영을 시작했으나, 곧바로 빈번한 수집 오류에 직면했습니다. 취하, 매각기일 변경 등 경매 사건 특유의 상태 변경을 예측하지 못해 테이블 간의 JOIN 관계가 깨지는 문제가 발생했습니다. 또한, 초기 설계 문서가 부재하여 팀원들에게 데이터 구조를 설명하고 문제를 추적하는 데에도 큰 어려움을 겪었습니다.

### Action (행동)
표면적인 에러 수정에 급급하기보다 근본적인 접근 방식을 바꾸기로 결단했습니다.
진행 중이던 구현을 멈추고 경매 사건의 라이프사이클(접수, 공고, 매각, 취하 등)을 문서화하며 도메인부터 다시 학습했습니다. 이를 바탕으로 '사건번호+법원+매각기일'을 복합 PK로 도입하여 상태 변경을 유일하게 식별하도록 테이블을 전면 재설계했습니다. 코드를 다시 짜기 전, DDL, ERD, 데이터 흐름도를 먼저 작성해 팀원들에게 공유하고 피드백을 받는 과정을 거쳤습니다.

### Result (결과 및 배운 점)
재설계 이후 수집 오류가 획기적으로 감소했고, IP 차단 대응 로직까지 안정적으로 태우며 일평균 3,000건의 데이터를 수집하는 체계를 완성했습니다. 분석가들이 별도의 JOIN 없이도 직관적으로 활용할 수 있는 비정규화 테이블 구조도 정착시켰습니다.
이 뼈아픈 실패를 통해 **"도메인 이해 → 설계 문서화 → 팀 공유 → 구현"**이라는 확고한 작업 원칙을 세웠습니다. 파이프라인의 생명은 속도가 아니라 '구조적 안정성'임을 깨달은 이 원칙은, 이후 진행된 모든 프로젝트를 재작업 없이 성공적으로 이끄는 밑거름이 되었습니다.

## 사례 2. 협업/소통 실패 — 일정 리스크 관리와 위기 극복: 폐쇄망 파견 현장에서의 상향식 소통

### Situation (상황)
검역본부 데이터 파이프라인 프로젝트 진행 당시, 최종 상용화를 위해 2주간 고객사의 폐쇄망(Closed Network) 환경에 파견되었습니다. 해당 기간 동안 기존에 담당하던 파이프라인을 포함하여 ASF(아프리카돼지열병), HPAI(조류인플루엔자), 실시간 차량 네트워크 등 총 3개의 주요 파이프라인 구축 및 안정화 작업을 완료해야 했습니다.

### Task (문제)
인터넷 연결이 제한된 폐쇄망 특성상, 예상치 못한 환경 변수와 에러 해결에 예상보다 많은 시간이 소요되었습니다. 한정된 2주라는 시간 안에 3개의 파이프라인을 병렬로 세팅하려다 보니 업무 집중도가 분산되었고, 자칫 기한 내에 단 하나의 파이프라인도 정상 가동하지 못할 리스크가 발생했습니다. 초기에는 주어진 업무를 모두 완수해야 한다는 주니어로서의 부담감에 혼자서 야근을 하며 문제를 해결하려 했습니다.

### Action (행동)
하지만 실무자가 이슈를 혼자 안고 가는 것이 프로젝트 전체 일정에 더 큰 지연을 초래할 수 있다고 판단했습니다. 즉시 병렬 작업을 중단하고, 각 파이프라인별 현재 진행률, 발생한 블로커(Blocker), 예상 소요 시간을 객관적인 문서로 정리했습니다. 이를 바탕으로 팀 리드에게 현재의 리스크를 투명하게 공유하고, 비즈니스 목적상 상용화가 가장 시급한 파이프라인이 무엇인지 우선순위 재조정을 요청하는 상향식 소통(Upward Communication)을 진행했습니다.

### Result (결과 및 배운 점)
팀 리더와의 조율을 통해 시급성이 가장 높은 질병 예측(ASF, HPAI) 파이프라인에 리소스를 먼저 집중하고, 차량 네트워크 분석은 후순위로 진행하는 것으로 일정을 재수립했습니다. 명확한 우선순위 하에 선택과 집중을 한 결과 작업에 속도가 붙었고, 무사히 2주 내에 3개의 파이프라인을 모두 안정화할 수 있었습니다.
이 경험을 통해 일정 압박이나 돌발 변수가 발생했을 때, 혼자서 무리하게 해결하려 하기보다 **'투명한 상황 공유와 대안 제시를 통한 리더와의 조율'**이 프로젝트를 지키는 가장 효과적이고 프로페셔널한 리스크 관리법임을 배웠습니다.
