## A. ETL / ELT / 파이프라인 기초

### Q1. ETL과 ELT의 차이는? 언제 어느 쪽을 쓰나요?
ETL은 데이터를 추출한 다음 외부에서 변환을 끝내고 저장하는 방식이고, ELT는 일단 데이터를 적재한 후에 저장소 내부에서 변환하는 방식입니다. 과거에는 데이터 저장소가 변환 비용을 감당하지 못해서 ETL이 일반적이었는데, 요즘은 BigQuery나 Snowflake 같은 분산형 데이터 웨어하우스가 등장하면서 ELT가 주류가 됐다고 알고 있습니다. 저는 실무에서 PostgreSQL 단일 DB 환경이었기 때문에 Python에서 변환을 끝내고 적재하는 ETL 방식을 주로 사용했습니다.

### Q2. 데이터 파이프라인에서 멱등성(Idempotency)이 왜 중요한가요?
**멱등성이라는 건** 같은 작업을 여러 번 실행해도 결과가 동일하게 나오는 성질을 말합니다. 예를 들어 같은 데이터를 두 번 적재해도 테이블 안에 중복이 안 생기고, 한 번 적재한 것과 결과가 똑같으면 멱등성이 보장된다고 합니다. 이게 중요한 이유는 데이터 파이프라인이 어떤 이유로든 실패해서 **재시도하거나 재실행할 일이 자주 생기기 때문**입니다. 멱등성이 없으면 재실행이 새로운 사고를 만들 수 있어서 운영이 위험해집니다.

저는 두 가지 방법을 주로 씁니다.
- **자연 키 기반 UPSERT**: `ON CONFLICT DO UPDATE`로 중복 적재를 방지
- **시점 기반 스냅샷**: `(키, collected_at)` 같은 복합 PK로 두면 같은 시점은 PK 충돌로 차단되고, 다른 시점은 별도 레코드로 누적

### Q3. Batch와 Streaming의 차이, 그리고 선택 기준은?
Batch는 일정 시간 동안 누적된 데이터를 한 번에 모아서 처리하는 방식이고, Streaming은 이벤트가 발생하는 즉시 처리하는 방식입니다. 선택 기준은 **레이턴시 요구사항**이라고 봅니다 — 분 단위 이상 지연이 허용되면 Batch, 초 단위 이하 응답이 필요하면 Streaming. 다만 Streaming은 운영 복잡도가 훨씬 높기 때문에, 정말 필요한 경우가 아니면 Batch로 시작하는 게 맞다고 생각합니다.

### Q4. exactly-once, at-least-once, at-most-once의 차이는?
메시지가 처리되는 횟수를 어디까지 보장할 것인지에 대한 수준입니다.
- **At-most-once**: 중복은 없지만 누락은 가능
- **At-least-once**: 누락은 없지만 중복은 가능
- **Exactly-once**: 둘 다 없음 (가장 이상적이지만 비용이 큼)

실무에서는 보통 **at-least-once + 멱등 처리**로 사실상의 exactly-once를 구현하는 패턴을 많이 씁니다. Kafka의 트랜잭션 API나 Flink의 checkpoint 같은 진짜 exactly-once는 인프라 비용이 크기 때문에, 정말 정합성이 중요한 도메인(예: 결제, 금융)에서만 쓰는 걸로 알고 있습니다.

### Q5. 데이터 파이프라인 설계 시 가장 중요하게 보는 것은?
저는 **실패 가시성과 재시작 용이성**을 가장 중요하게 봅니다. 테이블 단위로 task를 분리해서 장애가 났을 때 영향 범위를 좁히고, 어디서 어떤 이유로 실패했는지 로그를 명확하게 남겨서 원격에서도 복구가 가능하게 합니다. 검역본부 파이프라인이 폐쇄망에 원격 운영 환경이었기 때문에 이런 습관이 더 강화됐습니다.

## B. 데이터 모델링 / 스키마

### Q6. 정규화와 비정규화는 어떤 차이가 있고, 언제 어느 쪽을 선택하나요?
**정규화는** 중복을 제거하고 데이터를 여러 테이블로 쪼개서 저장하는 방식이고, **비정규화는** 의도적으로 중복을 허용해 한 테이블에 넓게 펴서 저장하는 방식입니다.

| 구분 | 장점 | 단점 |
|------|------|------|
| 정규화 | 저장 효율, 데이터 정합성 보장 | JOIN 비용 증가, 쿼리 복잡 |
| 비정규화 | JOIN 없이 빠른 조회 | 저장 공간↑, 갱신 시 중복 갱신 부담 |

원칙적으로 **트랜잭션이 많은 OLTP 환경은 정규화**, **분석 위주의 OLAP 환경은 비정규화**가 일반적입니다. 분석가가 매번 JOIN을 작성하는 게 비효율이거나, 동일 데이터를 반복 조회하는 패턴이라면 비정규화가 유리합니다.

### Q7. Primary Key와 Unique Index의 차이는?
- **PK**: 테이블당 하나만 가능, NULL 허용 안 됨, 자동으로 unique index 생성됨
- **Unique Index**: 여러 개 가능, NULL 허용 (NULL은 unique 비교에서 서로 다르게 취급)

개념적으로 PK는 "이 테이블에서 행을 식별하는 본질적인 키"이고, Unique Index는 "추가로 중복만 막고 싶은 컬럼"이라고 이해하면 좋습니다.

### Q8. Star Schema와 Snowflake Schema의 차이는?
**Star 스키마**는 중앙에 fact 테이블을 두고 그 주변에 비정규화된 dimension 테이블들이 평평하게 붙는 구조이고, **Snowflake 스키마**는 그 dimension을 다시 정규화해서 여러 단계로 가지치기되는 구조입니다.

| 구분 | Star | Snowflake |
|------|------|-----------|
| 차원 테이블 | 비정규화 | 정규화 |
| 저장 공간 | 더 많이 사용 | 절약 |
| 쿼리 성능 | 단순·빠름 | JOIN 많아 복잡 |
| 적합 | 데이터 마트 (단순 분석) | 데이터 웨어하우스 (대규모) |

분석용 DW에서는 보통 Star가 더 선호되는데, JOIN을 줄여서 분석 쿼리가 단순해지고 빨라지기 때문입니다. 다만 차원 정보가 자주 바뀌고 중복 갱신 부담이 크면 Snowflake로 가는 게 유지보수에 유리할 수 있습니다.

### Q9. SCD(Slowly Changing Dimension)에 대해 들어보셨나요?
솔직히 깊게 다뤄본 적은 없고 개념만 알고 있는 수준입니다. Dimension 테이블에서 값이 시간에 따라 바뀔 때 그 변화를 어떻게 보존할 것인지에 대한 패턴이고, Type 1은 **그냥 덮어쓰기**(이력 없음), Type 2는 **새 레코드를 추가해서 이력 보존**, Type 3은 **이전 값을 별도 컬럼에 보관**하는 방식 정도로 알고 있습니다. 제가 게임 챗봇에서 캐릭터 정보를 시점별 스냅샷으로 누적한 게 Type 2 비슷한 개념인 것 같습니다. 이 부분은 데이터 웨어하우스 모델링 영역이라 더 공부할 필요가 있다고 보고 있습니다.

### Q10. 파티셔닝(Partitioning)은 언제 어떻게 적용하나요?
**파티셔닝은** 테이블이 너무 커지면 하나의 큰 테이블을 여러 개의 작은 물리 파티션으로 나눠 저장하는 방식입니다. 보통 **월 단위나 일 단위 같은 시간 기준**으로 자르는 RANGE 파티셔닝을 가장 많이 씁니다. 저도 실무에서 적용했을 때는 주로 월 단위, 일 단위 파티션 테이블을 만들어서 운영했습니다.

이렇게 했을 때 얻는 구체적인 효과:
- **Partition Pruning**: WHERE 절에 파티션 키가 들어가면 해당 파티션만 스캔하고 나머지는 아예 안 봄. 스캔량이 극적으로 줄어 쿼리가 빨라짐
- **유지보수 분리**: 오래된 파티션만 통째로 DROP해서 데이터 정리 가능 (DELETE보다 훨씬 빠름)
- **인덱스 효율**: 파티션 단위로 인덱스가 작아져서 인덱스 깊이가 얕아짐
- **백업·복구 단위 분리**: 최근 파티션만 따로 백업하는 식의 운영 가능

다만 파티션 키가 WHERE 절에 안 들어오면 모든 파티션을 다 스캔해야 해서 오히려 느려질 수 있고, 파티션이 너무 많으면 plan 단계 비용이 커지는 단점이 있습니다.

## C. SQL / PostgreSQL 심화

### Q11. PostgreSQL 인덱스 종류는 어떤 게 있고, 각각 언제 쓰나요?
PostgreSQL은 인덱스 종류가 꽤 다양한데, 데이터 종류에 따라 적합한 인덱스가 다릅니다.

- **B-tree** — 가장 기본이 되는 인덱스. 등호(`=`), 부등호(`<`, `>`), 범위(`BETWEEN`), 정렬(`ORDER BY`) 모두 잘 먹습니다. 일반적인 컬럼은 거의 다 B-tree
- **Hash** — 등호(`=`) 비교 전용. WAL 로깅이 늦게 추가돼서 거의 안 쓰는 추세
- **GIN** (Generalized Inverted Index) — JSONB, 배열, 전문 검색(full-text)처럼 **하나의 컬럼 안에 여러 값이 들어 있는 경우**에 씁니다. 역색인(inverted index) 방식이라 "이 값이 어느 row에 있는가"를 빠르게 찾음
- **GiST** (Generalized Search Tree) — 공간 데이터(PostGIS), 범위 타입 같이 **기하학적·범위 비교**가 필요할 때 사용
- **BRIN** (Block Range Index) — 매우 큰 테이블에서 **컬럼 값이 물리적 저장 순서와 상관관계가 있을 때** (예: 시계열 created_at). 인덱스 크기가 매우 작은 게 장점
- **ivfflat / hnsw** — pgvector의 **벡터 유사도 검색**용 인덱스

저는 실무에서는 B-tree와 GIN(JSONB)을 주로 썼고, 검역본부 GIS 데이터에서는 GiST, 게임 챗봇에서는 hnsw를 사용했습니다.

### Q12. EXPLAIN ANALYZE는 어떻게 읽나요?
EXPLAIN ANALYZE는 위에서부터 트리 구조로 쓰여 있지만 실제 실행은 **안쪽(가장 깊은 노드)부터** 일어난다고 이해하면 됩니다. 제가 볼 때 핵심으로 보는 지표는 다음과 같습니다.

- **actual time** — 실제 실행 시간. 어디서 시간이 갑자기 튀는지 추적
- **rows** — planner가 추정한 row 수와 실제 row 수의 차이. 차이가 크면 통계가 outdated되어 있어서 plan이 잘못 짜진 신호
- **cost** — `cost=시작비용..총비용` 형태로 표시되는데, **하위 노드로 내려가면서 cost가 갑자기 폭증하는 지점**이 병목입니다. 거기가 인덱스가 안 타거나 잘못된 JOIN 방식을 선택한 부분
- **Seq Scan vs Index Scan** — Seq Scan이 떠 있으면 인덱스가 안 타는 상황. 의도한 거면 OK, 아니면 인덱스 점검 필요
- **JOIN 종류**: Nested Loop / Hash Join / Merge Join — 데이터 크기에 따라 적절한 방식이 다른데, 작은 테이블은 Nested Loop, 큰 테이블 간은 Hash Join이 일반적
- **Buffers** (BUFFERS 옵션 켜야 보임) — 메모리에서 읽은 hit과 디스크에서 읽은 read의 비율. read가 많으면 캐시 미스가 잦은 상태

게임 챗봇에서 JSONB 스캔을 Numeric 스캔으로 바꿔 5.3배 성능 개선했을 때도 EXPLAIN ANALYZE의 actual time과 buffer 차이로 검증했습니다.

### Q13. VACUUM과 ANALYZE의 차이는? 왜 이걸 하면 쿼리가 빨라지나요?
- **VACUUM** — 삭제·업데이트로 생긴 dead tuple을 정리해서 공간을 재사용 가능하게 만듦
- **VACUUM FULL** — 테이블을 통째로 다시 써서 디스크 공간까지 반환 (테이블 락이 잡혀서 위험)
- **ANALYZE** — planner가 사용할 통계 정보를 갱신

**왜 VACUUM ANALYZE 후에 쿼리가 빨라지냐면**, PostgreSQL은 **MVCC(Multi-Version Concurrency Control)** 방식이라 UPDATE를 할 때 기존 row를 수정하지 않고 **새 버전을 추가하고 옛 버전을 dead tuple로 표시**합니다. 이 dead tuple이 쌓이면:
1. **테이블이 실제 데이터보다 훨씬 부풀어** 있어서, 같은 row를 찾으려 해도 더 많은 페이지를 읽어야 함 (이걸 "table bloat"라고 부름)
2. **인덱스도 옛 버전을 가리키는 항목들로 부풀어** 있어서 인덱스 스캔 자체가 느려짐
3. **planner의 통계가 outdated**되어 있으면 잘못된 실행 계획이 짜져서 인덱스 안 타는 결정이 나옴

VACUUM이 dead tuple을 정리해주고, ANALYZE가 통계를 갱신해주면, ① 페이지 수가 줄어 I/O가 줄고, ② planner가 정확한 plan을 짤 수 있게 되어 쿼리가 다시 빨라지는 겁니다. autovacuum이 자동으로 돌긴 하는데, 큰 테이블은 임계값 도달이 늦어서 수동으로 트리거해야 할 때가 있습니다.

### Q14. JSONB와 JSON의 차이는? 그리고 정규화에 안 맞는데도 JSONB로 적재하는 이유는?
**기술적 차이부터 말씀드리면** JSON은 텍스트 그대로 저장하고(공백·키 순서 보존), JSONB는 바이너리로 파싱해서 저장합니다. 그래서 JSONB는 적재 시점에 살짝 느리지만 **읽기·인덱싱(GIN)·연산이 훨씬 빠르고**, 거의 모든 실무 케이스에서 JSONB를 씁니다.

**정규화에 안 맞는데도 JSONB로 적재하는 이유**는 여러 가지가 있습니다.
- **스키마가 자주 바뀌거나 가변적인 경우**: 외부 API 응답이 종종 필드가 추가·변경되는데, 이걸 매번 ALTER TABLE로 따라가면 마이그레이션 비용이 큼. JSONB로 받아두면 신규 필드도 일단 적재되고, 필요한 시점에 정규 컬럼으로 추출 가능
- **저장 후 분석 패턴이 미정**: 어떤 필드를 자주 쓸지 모르는 초기 단계에서는 통째로 저장
- **드물게만 쿼리되는 메타데이터**: 매 row마다 컬럼화하기엔 sparse한 정보들을 한 컬럼으로 모음

성능 차이는 케이스에 따라 다른데, **단순 키 조회**는 JSONB도 GIN 인덱스로 충분히 빠릅니다. 다만 **수치 비교나 집계 연산을 자주 하는 필드**는 JSONB 안에서 꺼내 쓰면 매번 파싱·캐스팅 비용이 들어서 일반 컬럼보다 느립니다. 저도 게임 챗봇에서 JSONB로 적재했다가, 자주 쓰는 수치를 일반 컬럼으로 평탄화(Flattening)해서 5배 가까운 성능 개선을 본 적이 있습니다. 결국 **자주 쓰는 핵심 필드는 정규 컬럼, 가변적이고 드물게 쓰는 부가 정보는 JSONB**로 가는 하이브리드 패턴이 실무에서 합리적입니다.

### Q15. CTE(WITH 절)와 서브쿼리의 차이는? 서브쿼리가 안 좋다고 배우는 이유는?
**문법적으로는** CTE는 이름을 붙여서 가독성을 높이는 표현 방식이고, 결과적으로는 서브쿼리와 비슷하게 동작합니다. PostgreSQL 12 이전에는 CTE가 항상 materialize(중간 결과 저장)되어 옵티마이저가 inline 최적화를 못 했지만, **12 이후로는 기본적으로 inline**되어 서브쿼리와 거의 동일하게 처리됩니다. 같은 CTE를 여러 번 참조해서 재계산을 피하고 싶을 땐 `WITH ... AS MATERIALIZED`로 강제할 수 있습니다.

**서브쿼리가 안 좋다고 배우는 이유**는 가독성 문제도 있지만 기술적인 부분이 더 큽니다.
- **상관 서브쿼리(Correlated Subquery)의 N+1 문제**: 바깥 쿼리의 각 row마다 안쪽 서브쿼리가 반복 실행될 수 있어서, row 수가 많으면 성능이 급격히 나빠짐
- **옵티마이저가 JOIN으로 변환하기 어려운 형태**: 의미상 JOIN으로 풀 수 있는 서브쿼리도 작성 방식에 따라 옵티마이저가 못 알아채면 그대로 비효율적으로 실행됨
- **EXPLAIN 보기 어려움**: 중첩된 서브쿼리는 실행 계획이 복잡해져서 디버깅 비용이 큼

그래서 실무에서는 **상관 서브쿼리를 JOIN이나 Window 함수로 풀어 쓰는 것**이 일반적입니다. CTE는 단순 가독성 측면에서도 도움이 되지만, 더 중요한 건 **복잡한 로직을 단계별로 분리해서 옵티마이저와 사람 모두에게 명확하게 전달**하는 효과가 있다는 점입니다.

## D. Airflow 심화

### Q16. Airflow의 핵심 컴포넌트 4가지는?
Airflow는 크게 네 개의 컴포넌트로 동작합니다.

- **Scheduler** — DAG 파일을 주기적으로 파싱해서 실행할 task를 metadata DB에 등록
- **Webserver** — UI를 제공. DAG 상태 모니터링, 로그 확인, 수동 트리거 등
- **Metadata Database** — DAG와 task의 상태, 설정 등을 저장 (보통 PostgreSQL이나 MySQL)
- **Executor / Worker** — Scheduler가 등록한 task를 실제로 실행하는 컴포넌트

흐름은 **Scheduler가 DAG 파싱 → 실행 시점이 된 task를 metadata DB에 큐잉 → Executor가 Worker에 분배 → Worker가 실행 후 결과를 다시 metadata DB에 기록** 이렇게 갑니다.

### Q17. Executor 종류와 각각의 동작 방식은?
Airflow는 운영 환경에 따라 여러 Executor 중 하나를 선택할 수 있습니다.

- **SequentialExecutor** — task를 1개씩 순차 실행. SQLite와 함께 개발용으로만 사용
- **LocalExecutor** — 같은 머신 안에서 병렬 실행. 단일 노드 운영에 적합. 저는 실무·개인 모두 이걸 Docker Compose 기반으로 사용했습니다
- **CeleryExecutor** — **Redis나 RabbitMQ 같은 메시지 큐**를 중간에 두고, Worker 노드를 여러 대 띄워서 task를 분산 실행. Scheduler가 task를 큐에 넣으면 여러 Worker 중 하나가 가져가서 실행하는 방식. 수평 확장이 쉬워서 대규모 운영에 일반적
- **KubernetesExecutor** — task마다 **Kubernetes Pod를 새로 띄워서 그 안에서 실행**하고, 끝나면 Pod를 제거. 격리성과 자원 효율이 좋고 task별 의존성을 다르게 가져갈 수 있다는 게 큰 장점이지만, K8s 클러스터 운영 부담이 따라옵니다

대규모 운영은 보통 CeleryExecutor 또는 KubernetesExecutor를 쓰는 걸로 알고 있고, 저는 아직 LocalExecutor 단계에서 더 위 단계는 학습 중인 영역입니다.

### Q18. XCom의 용도와 한계는?
**XCom은** task 사이에 작은 데이터를 전달하는 메커니즘입니다. 한계는 두 가지가 있습니다.
- 기본적으로 **metadata DB에 저장**되기 때문에 큰 데이터(>1MB)에는 부적합
- **직렬화 가능한 객체**만 전달 가능

큰 데이터는 XCom에 **경로나 포인터만 저장**하고 실제 데이터는 S3나 별도 DB에 두는 패턴이 일반적입니다. Airflow 2.5 이후로는 XComObjectStorage backend로 S3에 직접 저장하는 옵션도 생겼습니다.

### Q19. Sensor와 Pool은 각각 뭐고 언제 쓰나요?
**Sensor는** "외부 조건이 충족될 때까지 기다리는 task"입니다. 예를 들어 ① 특정 파일이 S3에 도착할 때까지, ② 다른 DAG의 task가 완료될 때까지, ③ 외부 API가 특정 응답을 줄 때까지 같은 식. 다만 Sensor가 worker slot을 점유한 채로 대기하면 자원 낭비가 크기 때문에 `mode='reschedule'` 옵션을 켜서 **주기적으로만 깨어나서 체크**하도록 운영하는 게 일반적입니다.

**Pool은** "동시에 실행될 수 있는 task의 개수를 제한하는 메커니즘"입니다. 예를 들어 ① 외부 API가 분당 100회 제한이 있어서 동시 호출을 5개로 묶고 싶을 때, ② DB connection 한도를 보호하고 싶을 때 사용합니다. Pool에 slot 수를 정해두고 task에 `pool='my_pool'`로 할당하면 그 풀 안에서만 동시성이 제한됩니다.

저는 TFT 프로젝트에서 Lambda 호출을 동시 3개로 제한할 때 Pool 개념을 활용했습니다.

### Q20. DAG 실패 시 재시도 전략은 어떤 게 있고, 각각 뭔가요?
Airflow에서 task 실패 시 적용할 수 있는 옵션이 여러 개 있는데, task 성격에 맞춰 차등 적용하는 게 중요합니다.

- **`retries`** — 실패 시 재시도할 횟수. 보통 2~3회 정도로 두는 게 일반적
- **`retry_delay`** — 재시도 사이 간격 (예: 5분 후 재시도)
- **`retry_exponential_backoff`** — True로 두면 재시도 간격을 점차 늘림 (1분 → 2분 → 4분). 외부 시스템 일시 장애에 유리
- **`max_retry_delay`** — 지수 backoff의 상한선
- **`on_failure_callback`** — 최종 실패 시 호출되는 콜백 함수. 저는 Discord webhook 알림 함수를 공통으로 등록해서 모든 DAG의 실패를 한 채널로 모았습니다
- **`execution_timeout`** — task가 무한 대기하는 걸 방지하는 timeout

적용 원칙은:
- **외부 API/네트워크 의존 task**는 일시 장애가 흔하니까 backoff retry 적용
- **순수 로직 오류**(예: 파싱 에러, 잘못된 SQL)는 재시도해도 같은 결과니까 `retries=0`
- **중복 실행이 위험한 task**(멱등성 없는 INSERT 등)는 retry를 신중하게 적용

### Q21. Backfill과 Catchup의 차이는?
둘 다 "과거 데이터를 처리한다"는 맥락이지만 작동 방식이 다릅니다.

- **Catchup (`catchup=True`)** — DAG의 `start_date`부터 현재까지 **누락된 모든 실행을 자동으로 한꺼번에 트리거**. 새 DAG 배포 시 의도치 않게 수십~수백 회 실행될 수 있어서 매우 위험
- **Backfill** — 사용자가 명시적으로 **`airflow dags backfill` 명령으로 특정 날짜 범위를 지정해서 실행**. Catchup의 안전한 버전이라고 보면 됩니다

저는 새 DAG는 항상 `catchup=False`로 두고 시작하고, 과거 데이터가 필요하면 명시적으로 backfill 명령을 사용합니다. Catchup은 정말 의도가 명확할 때만 켜는 게 안전합니다.

### Q22. Airflow의 logical date와 실제 실행 시간이 왜 다른가요?
Airflow에서 DAG의 **logical date(execution date)**와 **실제 task 실행 시간**은 다른 개념입니다.

- **logical date** — DAG이 "어떤 시점의 데이터를 처리해야 하는지"를 나타내는 논리적 시간. 데이터의 타임스탬프와 관련됨
- **실제 실행 시간** — Scheduler가 DAG을 실제로 트리거한 물리적 시간

예를 들어 DAG이 매일 오전 9시에 실행되도록 설정됐을 때, `2025-01-05 09:00:00`이라는 logical date는 사실 **2025-01-04 데이터를 처리하라는 의미**일 수 있습니다. 그리고 실제 task는 스케줄러 상태나 리소스 상황에 따라 9시가 아니라 9시 15분에 시작될 수도 있습니다. 이 차이를 모르면 데이터 시점과 실행 시점을 혼동해서 파이프라인 디버깅이 어려워질 수 있습니다.

## E. Big Data / 분산처리 / 아키텍처

### Q23. Hadoop의 HDFS와 MapReduce를 설명해주세요
**HDFS**는 큰 파일을 블록 단위(기본 128MB)로 쪼개서 여러 노드에 복제 저장하는 분산 파일시스템입니다. 같은 블록을 보통 3개의 노드에 복제해서 저장하는데, 노드 하나가 죽어도 다른 복제본이 있어서 데이터 손실 없이 운영이 가능한 구조입니다(이걸 fault tolerance라고 부릅니다).

**MapReduce**는 그 데이터에 대해 **Map(분할 처리) → Shuffle(키 기준 재배치) → Reduce(집계)** 단계로 분산 연산을 수행하는 프로그래밍 모델입니다. 각 단계 결과를 디스크에 쓰고 다음 단계가 다시 읽는 구조라 I/O 비용이 큽니다. 저는 대학에서 K-means, SON, PCY 알고리즘을 MapReduce로 직접 구현해본 경험이 있습니다.

### Q24. Spark가 Hadoop MapReduce보다 빠른 이유는?
가장 큰 차이는 **메모리 기반 처리**와 **lazy execution**입니다.
- MapReduce는 매 단계마다 결과를 디스크에 쓰고 다음 단계가 다시 읽는 구조라 I/O 비용이 큽니다
- Spark는 중간 결과를 **메모리(RAM)에 유지**하고, **DAG 기반 lazy execution**으로 전체 파이프라인을 보고 최적화한 후 한 번에 실행합니다

그래서 반복 작업(특히 ML)에서 10배 이상 빠른 경우가 많습니다. 다만 메모리가 부족하면 spill이 일어나면서 디스크를 쓰게 되어 성능 이점이 줄어드는 단점이 있습니다.

선택 기준은:
- **대규모 배치 처리, 안정성 우선** → Hadoop이 여전히 적합
- **반복 연산·실시간 분석·빠른 처리** → Spark가 더 적합

### Q25. Spark의 동작 방식과 핵심 컴포넌트는?
Spark는 클러스터 기반으로 동작하며 세 가지 핵심 컴포넌트로 구성됩니다.

- **Driver** — 애플리케이션의 진입점. 작업 계획(DAG)을 세우고 클러스터에 명령을 전달
- **Cluster Manager** — 클러스터 자원을 관리하고 작업 분배 (YARN, Kubernetes, Standalone 등)
- **Executor** — 실제 노드에서 task를 병렬 실행하고 결과를 Driver에 반환

작업 흐름:
1. 데이터를 로드해서 분산 데이터셋(RDD/DataFrame) 생성
2. **Transformation**으로 변환을 정의 (lazy — 즉시 실행 안 됨, DAG만 쌓임)
3. **Action**(collect, count, save 등)이 호출되면 비로소 DAG가 실행됨
4. Spark가 DAG를 분석해 Stage로 나누고, 각 Stage를 더 작은 Task로 쪼개서 Executor에 분배해 병렬 처리

이 lazy evaluation 덕분에 Spark가 전체 파이프라인을 보고 predicate pushdown, projection pruning 같은 최적화를 적용할 수 있습니다.

### Q26. Kafka의 핵심 개념과 동작 구조는?
Kafka는 다음 구성으로 작동하는 분산 메시징 시스템입니다.

- **Producer** — 데이터를 특정 Topic으로 전송
- **Broker** — 데이터를 받아서 Topic의 Partition에 저장
- **Topic** — 메시지의 카테고리 (테이블 같은 개념)
- **Partition** — Topic을 나눈 단위. 병렬 처리와 ordering 보장의 단위
- **Consumer** — 특정 Topic을 구독해서 Partition에서 데이터를 읽음
- **Consumer Group** — 같은 group에 속한 consumer들이 partition을 나눠 읽음 (수평 확장)
- **Zookeeper / KRaft** — 클러스터 상태와 메타데이터 관리

핵심 특성으로, **같은 partition 안에서는 순서가 보장되지만 partition 간 순서는 보장되지 않습니다**. 그래서 순서가 중요한 메시지는 **같은 키를 같은 partition에 보내는 키 기반 partitioning**으로 처리합니다. 저는 대학 과제에서 Spark Structured Streaming으로 Kafka에서 트위터 데이터를 받아 처리해본 경험이 있고, 실무 경험은 없습니다.

### Q27. CAP 정리란 무엇인가요?
CAP 정리는 분산 시스템에서 다음 세 가지를 **동시에 모두 만족할 수 없다**는 이론입니다.

- **C — Consistency(일관성)**: 모든 노드가 같은 시점에 같은 데이터를 보여줌
- **A — Availability(가용성)**: 모든 요청이 응답을 받음 (성공이든 실패든)
- **P — Partition Tolerance(분할 내성)**: 네트워크가 끊겨도 시스템이 계속 동작

네트워크 분할(P)은 현실에서 피할 수 없기 때문에, 실질적인 선택은 **CP냐 AP냐**가 됩니다. 시스템별 분류 예시:

- **CP 계열** (일관성 우선)
  - **RDB** (PostgreSQL, MySQL): 트랜잭션 ACID 보장. 네트워크 문제 시 응답 거부
  - **HBase**: Hadoop 기반 분산 NoSQL. 컬럼 패밀리 모델, 강한 일관성
- **AP 계열** (가용성 우선)
  - **Cassandra**: Wide-column NoSQL. 분산 환경에서 항상 응답하지만 일관성은 eventual
  - **DynamoDB**: AWS의 매니지드 NoSQL. 기본 eventual consistency, 옵션으로 strong consistency 선택 가능

다만 실제 시스템은 이분법적이지 않고, **trade-off의 스펙트럼** 위에 있다고 보는 게 맞습니다.

### Q28. Consistent Hashing(해시 링)에 대해 아시나요?
**Consistent Hashing은** 분산 시스템에서 데이터를 노드들에 나눠 저장할 때, **노드의 추가나 삭제가 있어도 기존 데이터의 이동을 최소화**하는 기법입니다.

동작 원리:
1. 데이터와 서버 노드를 해시 함수로 변환해서 **해시 링** 위에 배치
2. 데이터는 자기 해시값 기준으로 **시계 방향으로 가장 가까운 노드**에 저장됨
3. 새 노드를 추가하면 그 노드가 담당할 일부 데이터만 이웃 노드에서 가져옴 (전체 재배치 X)
4. 노드 삭제도 비슷하게 영향 범위가 최소화됨

장점:
- 노드 추가·삭제 시 **데이터 이동이 최소화**됨 (전통적 해시는 노드 수가 바뀌면 거의 전체 재해싱 필요)
- 균형 잡힌 데이터 분배 가능

단점:
- 항상 균등 분배는 보장 안 됨 → 데이터 쏠림(skew) 발생 가능
- 해결 방법: **Virtual Node**(하나의 물리 노드를 여러 가상 노드로 쪼개서 링에 배치)를 쓰면 분산이 균등해짐. 단 메모리 추가 사용 trade-off가 있음

DynamoDB, Cassandra, 분산 캐시(Memcached, Redis Cluster) 등에서 많이 사용되는 핵심 알고리즘입니다.

### Q29. 빅데이터 아키텍처 종류(Batch / Lambda / Kappa)에 대해 설명해주세요
크게 세 가지 아키텍처 패턴이 있습니다.

**1. 배치 아키텍처 (Batch Architecture)**
- 데이터를 일정 시간 간격으로 모아서 한 번에 처리
- 대량 데이터의 안정적 처리에 강하고, 배치 최적화 시스템에 효율적
- 단점: 실시간 처리 부적합, 데이터가 누적된 후에만 결과 확인 가능

**2. 람다 아키텍처 (Lambda Architecture)**
- 실시간 스트리밍과 배치 처리의 장점을 결합
- 두 개의 독립적인 레이어로 운영
  - **Batch Layer**: 정확하고 완전한 데이터 처리
  - **Speed Layer**: 실시간 근사 처리
  - **Serving Layer**: 두 결과를 합쳐서 사용자에게 제공
- 단점: 같은 로직을 두 레이어에 따로 구현해야 해서 운영 복잡도가 큼

**3. 카파 아키텍처 (Kappa Architecture)**
- 람다의 단점을 해결하기 위해 **배치 계층을 제거하고 모든 처리를 스트림으로 통일**
- 시스템이 단순해지고 코드 중복도 사라짐
- 단점: 대규모 배치 처리에 부적합, 과거 데이터 재처리가 까다로움

요즘은 **Lakehouse(Delta Lake, Iceberg)** 같은 새로운 패턴이 나오면서 배치와 스트리밍을 같은 저장소에서 다루는 방향으로 진화하고 있다고 알고 있습니다.

### Q30. ACID에 대해 각 요소를 설명해주세요
ACID는 트랜잭션이 안전하게 처리되기 위한 네 가지 속성입니다.

- **A — Atomicity(원자성)**: 트랜잭션 안의 모든 작업이 **전부 성공하거나 전부 실패**해야 함. 부분 성공은 없음
- **C — Consistency(일관성)**: 트랜잭션 전후로 DB가 정의된 제약(무결성, FK 등)을 항상 만족
- **I — Isolation(격리성)**: 동시에 실행되는 트랜잭션끼리 서로 영향을 주지 않음. 격리 수준에 따라 보장 정도가 다름 (Read Uncommitted, Read Committed, Repeatable Read, Serializable)
- **D — Durability(지속성)**: 커밋된 트랜잭션의 결과는 시스템이 죽어도 영구적으로 보존됨 (보통 WAL 로그를 디스크에 쓰는 방식으로 보장)

PostgreSQL을 비롯한 RDB는 ACID를 강하게 보장하고, NoSQL은 일부를 완화해서 가용성·확장성을 얻는 trade-off가 있습니다.

### Q31. 인덱스의 장단점과 선택 기준은?
**인덱스의 장점:**
- **쿼리 성능 향상**: WHERE, JOIN, ORDER BY에서 빠른 조회
- **제약 조건 강제**: Unique 인덱스로 중복 방지
- **정렬·그룹화 가속**: 인덱스가 정렬된 상태로 저장되어 있어 ORDER BY가 빨라짐

**인덱스의 단점:**
- **저장 공간 추가 필요**
- **INSERT/UPDATE 성능 저하**: 데이터 변경 시 인덱스도 함께 갱신해야 함
- **오버 인덱싱**: 너무 많이 만들면 오히려 plan 단계 비용이 커지고, 쓰기 성능이 크게 나빠짐

**어떤 컬럼에 인덱스를 만들면 좋은가:**
- WHERE, JOIN 조건에 자주 등장하는 컬럼
- **카디널리티(Cardinality)가 높은 컬럼** — 즉 유니크한 값이 많은 컬럼
- **선택도(Selectivity)가 좋은 컬럼** — 인덱스 한 번에 추려지는 row 비율이 작을수록 효율적

성별처럼 카디널리티가 낮은 컬럼은 인덱스가 비효율적이고, 사용자 ID처럼 거의 유니크한 컬럼은 인덱스가 매우 효율적입니다.

### Q32. B+ Tree 인덱스의 특징과 시간복잡도는?
**B+ Tree는** 균형 잡힌 다중 자식 트리로, RDB 인덱스의 사실상 표준 자료구조입니다.

특징:
- **내부 노드는 키만 저장**하고, **실제 데이터는 리프 노드에만 저장**
- **리프 노드들이 서로 연결(Linked List)**되어 있어서 범위 쿼리(`BETWEEN`, `<`, `>`) 효율이 매우 좋음
- 트리의 균형이 항상 유지되어 검색 성능이 일정함
- 디스크 I/O가 중요한 환경에서 우수한 성능

시간복잡도:
- 검색·삽입·삭제: **O(log N)**
- 범위 쿼리: **O(log N + M)** (M은 결과 row 수)

B-Tree와의 차이점은 **B-Tree는 내부 노드에도 데이터를 저장**하지만 B+ Tree는 리프에만 저장한다는 점입니다. B+ Tree가 메모리 효율과 범위 검색에서 유리해서 RDB 인덱스로 더 많이 쓰입니다.

### Q33. ElasticSearch는 어떻게 동작하나요?
ElasticSearch는 **대규모 데이터를 빠르게 검색·분석**할 수 있는 분산형 검색 엔진입니다.

핵심 특성:
- **문서 기반 저장**: JSON 형태의 문서를 저장
- **RESTful API**: HTTP로 데이터 조작 가능
- **샤딩(Sharding)**: 대용량 데이터를 여러 노드에 분산 저장
- **복제(Replication)**: 고가용성 보장
- **집계 기능**: 분석용 aggregation 지원
- **실시간 검색**: 적재 후 거의 즉시 검색 가능 (로그 분석, 사이트 검색에 자주 사용)

핵심 동작 원리는 **Inverted Index(역색인)** 입니다. 각 단어를 키로 사용하고, 그 단어가 등장하는 문서 ID들을 매핑해서 저장하는 방식이라, 검색 키워드가 들어오면 해당 단어의 문서 ID 목록을 바로 가져올 수 있어 대규모 데이터에서도 빠른 검색이 가능합니다.

## F. 데이터 품질 / 신뢰성

### Q34. 데이터 품질은 어떻게 검증하나요?
**개념적으로** 데이터 품질 검증은 다음 영역으로 나뉩니다.
- **완전성(Completeness)**: NULL이나 누락된 row가 없는가
- **정확성(Accuracy)**: 값이 도메인 규칙을 만족하는가 (음수가 안 되는 컬럼에 음수가 없는지 등)
- **일관성(Consistency)**: 테이블 간 참조 무결성, 같은 의미의 값이 같게 표현되는가
- **유일성(Uniqueness)**: PK·자연키의 중복이 없는가
- **시의성(Timeliness)**: 적시에 최신화되는가

저는 실무에서는 **각 수치 컬럼에 대한 기초 통계 정보**(row 수, null 비율, min·max·평균) 정도의 검수를 자체 Python으로 만들어서 운영했습니다. 부동산 파이프라인에서는 격자-시세 미매핑이나 격자-PNU 미매핑 같은 결합도(join key 매칭률)도 자동 탐지·수정하는 검수 task를 만들었습니다.

표준 도구로는 **Great Expectations**나 **dbt test** 같은 게 있는데, 직접 도입한 적은 없고 학습 중인 영역입니다.

### Q35. 데이터 lineage(계보) 추적이 왜 중요하고, 어떤 도구를 쓰나요?
**lineage는** 어떤 데이터가 어디서 와서 어떤 변환을 거쳐 어디로 가는지를 추적하는 개념입니다. 중요한 이유:
- **장애 시 영향 범위 파악** — 원천 테이블이 깨졌을 때 어느 다운스트림이 영향받는지
- **신뢰도 검증** — 분석가가 보는 숫자가 어디서 어떻게 계산됐는지 추적 가능
- **컴플라이언스** — 개인정보 같은 민감 데이터의 흐름을 감사

대표적인 도구들:
- **dbt**: SQL 기반 데이터 변환 도구로, 변환 로직을 모델로 정의하면 의존성을 자동으로 그려주고 lineage 그래프를 제공
- **Marquez**: 오픈소스 lineage 추적 도구, OpenLineage 표준 기반
- **DataHub**: LinkedIn에서 만든 메타데이터·lineage 통합 카탈로그
- **Apache Atlas**: Hadoop 생태계의 메타데이터·lineage 관리

요즘은 **OpenLineage**라는 표준이 자리잡으면서 도구 간 연동이 쉬워지는 방향으로 가고 있습니다. 저는 실무에서 Airflow의 task 의존성과 코드로 사실상 lineage 역할을 했고, 표준 도구는 학습이 필요한 영역입니다.

### Q36. SLA / SLO / SLI 차이는?
- **SLI (Indicator)** — 측정 가능한 지표 (성공률, 지연시간 등)
- **SLO (Objective)** — SLI의 목표치 (예: 성공률 99.9%)
- **SLA (Agreement)** — SLO를 못 지켰을 때의 책임을 명시한 계약

데이터 파이프라인 예시로 보면:
- "매일 오전 9시까지 어제 데이터 적재 완료" → **SLA**
- "9시까지 완료율 99%" → **SLO**
- "실제 9시까지 완료한 비율" → **SLI**

## G. AI / LLM / RAG 이론 (게임 챗봇 프로젝트 기반)

### Q37. RAG(Retrieval-Augmented Generation)가 무엇이고 왜 쓰나요?
**RAG는** LLM이 답변하기 전에 관련 문서를 외부에서 검색해서 프롬프트에 포함시키는 패턴입니다. 쓰는 이유는 세 가지 정도입니다.
- LLM은 학습 시점 이후의 정보를 모름 (예: 최근 게임 패치)
- 도메인 특화 데이터(특정 회사 내부 데이터, 특정 게임 정보)는 학습되어 있지 않음
- Fine-tuning보다 비용이 훨씬 낮고 정보 갱신이 즉시 가능

저는 게임 챗봇에서 스키마 메타데이터와 few-shot 예제를 임베딩해서, 사용자 질의와 유사한 것만 LLM에 주입하는 방식으로 RAG를 활용했습니다.

### Q38. Embedding(임베딩)이 무엇이고, 왜 벡터로 변환해서 쓰나요?
**임베딩은** 텍스트를 의미를 보존하는 **고차원 벡터(보통 768~3072차원)** 로 변환한 표현입니다.

**왜 벡터로 변환해서 쓰는가** 하면, 컴퓨터는 텍스트의 의미를 직접 비교할 수가 없습니다. "강아지"와 "개"는 의미가 비슷하다는 걸 문자열 매칭으로는 절대 알 수 없죠. 그런데 임베딩으로 변환하면:
- 의미가 비슷한 문장은 **벡터 공간에서 가까운 위치**로 매핑됨
- 거리(코사인 유사도, L2 거리)만 계산하면 의미 유사도를 정량적으로 측정 가능
- 키워드가 다르더라도 의미가 같으면 검색 가능 (예: "환불 절차" 질의 → "반품 안내" 문서 검색됨)

기술적 효과:
- **검색 정확도 향상** — 키워드 매칭 한계를 넘어선 의미 기반 검색
- **언어·표현 차이 흡수** — 다른 단어, 다른 어순도 같은 의미면 가까이 매핑
- **분류·클러스터링 가능** — 비슷한 의미의 문서들을 자동으로 군집화 가능

저는 OpenAI의 임베딩 모델을 게임 챗봇에서 사용했습니다.

### Q39. Vector Similarity의 종류와 차이는?
주로 세 가지를 씁니다.
- **Cosine Similarity**: 두 벡터 사이 **각도**만 보고 크기는 무시. 텍스트 임베딩에 가장 많이 사용
- **L2 Distance(유클리드)**: 두 벡터 끝점 사이 **직선 거리**. 크기까지 반영
- **Inner Product(내적)**: 두 벡터를 곱한 합. 정규화된 벡터에서는 cosine과 동등

OpenAI 임베딩처럼 이미 정규화돼 있는 모델은 셋 다 결과가 거의 같지만, 일반적으로는 cosine을 기본값으로 사용하는 게 안전합니다.

### Q40. Vector DB의 종류와 pgvector가 Pinecone과 다른 점은?
주요 Vector DB:
- **pgvector**: PostgreSQL 확장. RDB와 같은 DB 안에서 벡터 검색을 지원. 운영이 단순하고 RDB JOIN과 결합 가능
- **Pinecone**: 매니지드 SaaS. 대규모 벡터 데이터에 최적화된 전용 인프라, 운영 부담이 거의 없음
- **Weaviate / Qdrant**: 자체 호스팅 가능한 전용 vector DB, 고급 검색 기능 풍부

**pgvector vs Pinecone 차이점**:
| 측면 | pgvector | Pinecone |
|------|----------|----------|
| 운영 | 직접 관리 (PostgreSQL과 함께) | 매니지드 (운영 부담 X) |
| 비용 | PostgreSQL 인스턴스 비용 | 사용량 기반 SaaS 요금 |
| 규모 | ~1억 row 정도까지 적합 | 수십억 row 대규모 가능 |
| RDB 통합 | 한 DB에서 JOIN 가능 | 별도 시스템, 동기화 필요 |
| 검색 성능 | 데이터 규모 커지면 한계 | 대규모에서도 안정적 빠름 |
| 벤더 lock-in | 없음 (PostgreSQL 표준) | 있음 |

저는 게임 챗봇에서 pgvector를 선택했는데, 게임 메타데이터(RDB)와 임베딩(Vector)을 한 DB에서 JOIN할 수 있어 인프라·정합성 비용이 두 배로 늘어나는 걸 피하고 싶었기 때문입니다. 데이터 규모가 수억 row를 넘어가면 Pinecone이나 Qdrant 같은 전용 솔루션으로 옮겨야 한다고 생각합니다.

### Q41. HNSW와 IVFFlat 인덱스는 어떻게 다른가요?
둘 다 벡터 유사도 검색을 빠르게 하기 위한 ANN(Approximate Nearest Neighbor) 인덱스입니다.

- **IVFFlat (Inverted File with Flat Compression)**: 벡터들을 미리 클러스터링해두고, 검색 시 **가까운 클러스터 몇 개만 탐색**. 학습 단계가 필요하고 메모리 효율은 좋지만 정확도와 속도는 HNSW보다 살짝 낮음
- **HNSW (Hierarchical Navigable Small World)**: 그래프 기반 다층 구조로, 위에서부터 점점 정밀하게 내려가며 탐색. **검색 속도와 정확도 모두 우수**하지만 메모리 사용량이 큼

대용량·낮은 latency가 중요하면 HNSW, 메모리·인덱스 빌드 비용이 중요하면 IVFFlat을 씁니다. 저는 게임 챗봇에서 HNSW를 사용했습니다.

### Q42. Text-to-SQL의 동작 원리와 어려운 점은?
**Text-to-SQL은** 사용자의 자연어 질의를 LLM이 DB 스키마와 함께 보고 SQL을 생성하는 패턴입니다. 어려운 점은 세 가지:
- **할루시네이션** — 없는 테이블이나 컬럼을 만들어내서 SQL 생성
- **토큰 비용 폭증** — 스키마가 크면 프롬프트가 너무 길어져 비용 증가
- **모호한 질의** — "최근"이 며칠인지, "인기"의 기준이 무엇인지 같은 모호함

저는 게임 챗봇에서 토큰 비용 문제를 해결하기 위해 **Vector Search로 관련 스키마만 선택해서 주입**하는 4단계 진화 과정을 거쳐 토큰 84% 절감 + 정확도 100%를 달성했습니다.

### Q43. LLM의 Hallucination(환각) 대응 방법은?
환각을 줄이는 대표적인 방법들:
- **RAG**: 답변의 근거를 외부 데이터에서 가져오게 함 (가장 효과적)
- **Few-shot 예시**: 정답 형식의 예시를 프롬프트에 미리 보여줌
- **Output 제약**: function calling이나 JSON schema로 형식을 강제
- **Self-consistency**: 같은 질문을 여러 번 돌려서 일치 여부 검증
- **Validation Layer**: 생성된 결과(예: SQL)가 실제 스키마와 맞는지 코드로 사전 검증

저는 Text-to-SQL에서 **Vector Search + Few-shot + 실행 전 SQL 검증** 조합으로 할루시네이션을 줄였습니다.

### Q44. Token과 Context Window가 무엇인가요?
- **Token** — LLM이 처리하는 텍스트의 기본 단위. 한국어는 보통 1글자가 1~3 token, 영어는 1단어가 1 token 내외
- **Context Window** — 한 번의 호출에 입력 가능한 최대 token 수 (예: GPT-4 turbo 128k, Gemini 2M)

Context window가 클수록 더 많은 정보를 한 번에 처리할 수 있지만, ① **비용은 token 수에 비례**, ② **"lost in the middle"**라고 해서 긴 컨텍스트의 중간에 있는 정보를 놓치는 현상이 있어서 무작정 큰 게 좋은 건 아닙니다.

### Q45. Streaming response는 어떻게 동작하나요?
LLM이 토큰을 **하나씩 생성하는 즉시 사용자에게 SSE(Server-Sent Events)로 전송**하는 방식입니다. 전체 응답이 완성될 때까지 기다리지 않으니까 체감 응답성이 훨씬 좋습니다. 저는 포트폴리오 챗봇에서 `text/event-stream` Content-Type으로 `data: {content}\n\n` 형식으로 흘려보내고, 클라이언트가 점진적으로 렌더링하는 구조로 구현했습니다.

## H. 자주 받는 추가 질문 (과거 면접 복기 기반)

### Q46. 데이터 엔지니어와 데이터 아키텍트의 차이는?
**데이터 엔지니어는** 실제로 데이터 파이프라인을 설계·구축·운영하는 실무 역할입니다. 코드를 작성하고, 스케줄러를 운영하고, 데이터 적재·변환·정합성을 책임집니다.

**데이터 아키텍트는** 더 상위 레벨에서 **조직 전체의 데이터 구조와 흐름을 설계**하는 역할입니다. 어떤 시스템을 어떻게 연결할지, 어떤 저장소를 쓸지, 거버넌스·표준은 어떻게 가져갈지 같은 큰 그림을 그립니다.

요약하자면 데이터 아키텍트는 "무엇을 어떤 방향으로 만들 것인지", 데이터 엔지니어는 "그 방향을 실제 시스템으로 구현하는 사람" 정도로 이해할 수 있습니다.

### Q47. SparkSQL을 왜 사용하셨나요? 다른 대안은?
TFT 프로젝트에서 SparkSQL을 사용한 이유는, **SQL과 거의 비슷한 문법**으로 분산 처리를 할 수 있어서 학습 비용이 낮고, Spark의 메모리 기반 연산을 그대로 활용할 수 있어서 빠른 데이터 처리가 가능했기 때문입니다.

대안으로는 **Amazon Athena**도 고려해볼 수 있습니다. Athena는 S3에 있는 데이터를 별도의 로딩 없이 SQL로 바로 쿼리할 수 있다는 큰 장점이 있어서, 클라우드 환경이라면 더 매력적인 선택지일 수 있습니다.

다만 솔직히 말씀드리면 TFT 프로젝트는 데이터 규모상 PostgreSQL 집계로도 충분했는데, **분산 처리 학습 경험을 쌓고 싶어서 SparkSQL을 도입한 측면**이 컸습니다. 이걸 통해 도구 선택 시 학습 욕구와 실제 필요를 분리해서 판단해야 한다는 걸 배웠습니다.

### Q48. Airflow에서 어떤 Operator를 주로 썼고, 왜 썼나요?
저는 주로 다음 Operator들을 사용했습니다.
- **PythonOperator** — 가장 많이 사용. Python 함수로 작성한 ETL 로직을 그대로 실행
- **PostgresOperator** — DB에 직접 SQL을 실행할 때 (DDL, 단순 INSERT 등)
- **BashOperator** — 외부 스크립트나 CLI 도구 실행 (Selenium 크롤러 트리거 등)
- **TriggerDagRunOperator** — DAG 간 의존성 연결 (원천 수집 DAG 완료 후 산출 DAG 트리거)
- **ExternalTaskSensor** — 다른 DAG의 특정 task가 완료될 때까지 대기
- **BranchPythonOperator** — 조건에 따라 분기 (예: 평일/주말 다른 처리)

PythonOperator를 가장 많이 쓴 이유는 **ETL 로직을 일반 Python 코드로 짜는 게 가장 자유도와 유지보수성이 좋기** 때문입니다.

### Q49. HDFS는 어떻게 fault tolerance를 보장하나요?
HDFS는 **블록 복제(Block Replication)** 메커니즘으로 fault tolerance를 보장합니다.

동작 방식:
1. 큰 파일을 128MB 같은 블록 단위로 쪼갬
2. 각 블록을 **기본 3개의 노드에 복제**해서 저장
3. 복제본 배치 전략은 보통 ① 같은 랙에 2개, ② 다른 랙에 1개를 둬서 랙 단위 장애에도 대응
4. NameNode가 모든 블록의 위치를 메타데이터로 관리
5. 노드가 죽으면 NameNode가 감지하고, 복제본 수가 부족해진 블록을 **다른 노드에 자동으로 재복제**

이 구조 덕분에 노드 하나가 죽어도 데이터 손실이 없고, 자동 복구까지 이뤄집니다. 다만 NameNode 자체가 단일 장애점이 될 수 있어서 HA 구성을 별도로 가져가는 게 일반적입니다.

### Q50. SQL에서 `RANK() OVER (PARTITION BY ...)`와 `ROW_NUMBER()`의 차이는?
둘 다 윈도우 함수로 순위를 매길 때 사용하지만, **동률 처리 방식**이 다릅니다.

- **`ROW_NUMBER()`**: 같은 값이어도 무조건 1, 2, 3, 4, ... 순으로 유니크한 번호 부여
- **`RANK()`**: 같은 값은 같은 순위를 받고, 다음 순위는 건너뜀 (1, 2, 2, 4)
- **`DENSE_RANK()`**: 같은 값은 같은 순위, 다음 순위는 안 건너뜀 (1, 2, 2, 3)

선택 기준:
- "각 그룹에서 1등만 뽑고 싶다, 동률은 임의로 하나" → `ROW_NUMBER()`
- "동률이면 모두 같은 순위로 처리" → `RANK()` 또는 `DENSE_RANK()`

저는 게임 챗봇에서 캐릭터 장비 시점별 변화를 추적할 때 `ROW_NUMBER()`로 가장 최근 스냅샷만 뽑는 패턴을 자주 사용합니다.
