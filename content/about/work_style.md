# 일하는 방식과 협업 스타일

> "어떤 스타일로 일하시나요?" / "팀에서 어떤 역할을 맡나요?" 질문 대비.
> 가능한 항목마다 **짧은 근거 경험 1줄**을 곁들여 추상적 표현을 피함.

---

## 일하는 방식의 원칙

### 도메인 이해 → 설계 문서화 → 팀 공유 → 구현
경매 데이터 파이프라인 초기에 "일단 동작하게 만들자"로 접근했다가 도메인 특성(사건 상태 변경, 취하·기일변경 등)을 충분히 파악하지 못해 수집 오류가 반복된 이후 확립된 순서. 도메인 질문을 먼저, 설계 문서화 후 구현을 시작합니다.

### 근거 있는 해결책
현상만 고치지 않고 원인을 분석한 후 대응합니다. **IP 차단 이슈를 단순 프록시 추가가 아닌 "수집 대상 최적화"(일 12,000건 → 4,000건 타겟팅)**로 해결한 것이 대표 예시. 표면 대응은 결국 반복된다는 것을 실무에서 체감했습니다.

### 문서화를 설계의 일부로 취급
테이블 설계를 문서화 없이 진행해서 팀원 설명에 어려움을 겪은 이후, **DDL·ERD·데이터 흐름도 작성을 구현보다 먼저** 합니다. 작성 도중에도 핵심 결정 포인트는 팀에 노출하는 습관.

### 작업 단위를 작게 쪼개기
1주 단위 큰 작업보다 1~2일 단위로 쪼개서 진행 상황을 자주 가시화합니다. 추정 시간이 항상 실제보다 짧아진다는 것을 인정하고, 쪼개진 단위마다 검증·공유.

## 커뮤니케이션 스타일

### 비개발 직군과의 소통
검역본부 공무원, 부동산 고객사 등 **비개발 직군이 최종 사용자**였던 경험이 많음. 기술 용어 대신 업무 맥락에 맞는 언어로 설명하는 습관이 있고, 데이터 흐름을 그림으로 그려 공유하는 것을 기본으로 합니다.

### 다문화 환경에서의 조율
홍콩중문대학교 재학 중 다양한 국적·언어 배경의 팀원들과 프로젝트를 수행하며, 관점이 다를 때 **의견 대립보다 공통 목표 재확인**으로 조율하는 방식을 익혔습니다.

예시 방향:
- **"상대 입장을 먼저 본인 언어로 정리해서 오해가 없는지 확인"** — 의견 차이의 절반은 사실 오해이거나 trade-off 우선순위 차이
- **"데이터·근거 기반으로 논거 제시"** — 감정 대립보다 측정 가능한 근거로 토론

- 홍콩 다문화 팀워크 경험을 자연스럽게 연결 가능

## 학습 방식

### 독학 경험
AWS Cloud, Airflow는 **실무 투입 전 독학으로 기초를 다졌습니다**. 공식 문서를 1차 자료로 삼고, 실제 간단한 토이 프로젝트를 만들어보며 개념을 체화하는 방식을 선호합니다.

### 학습 기록
블로그에 학습 내용을 정리하는 것을 중요하게 생각함 (본 포트폴리오 사이트의 blog 섹션). **외부에 공개 가능한 수준으로 정리해야 스스로도 이해한 것**으로 보는 기준이 있습니다.

## 코드 품질 기준

### 주석보다 네이밍
**왜 이렇게 짰는지가 비자명할 때만 주석을 답니다**. 함수·변수명으로 의도를 드러내는 것을 우선. 주석은 코드가 변경되면 따라 변경되지 않아 거짓이 되기 쉽기 때문.

### 추상화는 필요할 때만
**세 번 반복되기 전까지는 공통화하지 않습니다**. 실무에서 섣부른 공통화로 유지보수가 어려워진 코드를 본 경험 이후 형성된 기준. 두 곳에서 비슷해 보여도 도메인 맥락이 다르면 별도로 두는 게 낫다는 판단.

### 코드 리뷰 시 가장 신경 쓰는 것
[수정 필요: 본인 우선순위]

예시 방향:
- **"이름이 의도를 드러내는지"** — 함수명만 보고 무엇을 하는지 명확한가
- **"엣지 케이스 처리"** — null, 빈 배열, 타임아웃, 재시도 시나리오
- **"트랜잭션·멱등성 범위"** — 데이터 파이프라인에서 특히 중요
- **"테스트 가능한 구조"** — 외부 의존성 분리, 순수 함수화 가능성

## 강점과 성장 포인트

### 강점
- **도메인 파고들기**: 사용자(공무원·고객사·분석가)의 업무 흐름을 파악해서 그들이 실제 쓰는 언어로 설계
- **문제 원인 분석**: 표면 현상이 아닌 근본 원인을 찾아 구조적으로 해결
- **자율 학습**: 투입 전에 필요한 스택을 스스로 학습해서 온보딩 속도 단축

### 성장 중인 부분
- **대규모·실시간 처리 경험**: 실무가 일 배치 중심이라 Spark·Kafka 기반 실시간 처리는 학습·개인 프로젝트 단계
- **테스트 커버리지·CI/CD 도입**: 실무에서는 검수 task 위주였고, 표준 도구(dbt test, pytest, GitHub Actions for DAG) 도입 경험 부족
- **플랫폼 관점 설계**: 개별 DAG 단위는 익숙하지만, 메타데이터·카탈로그·권한까지 포함한 플랫폼 단위 설계는 다음 학습 영역
- [수정 필요: 최근 의식적으로 개선하고 있는 습관]
  예: "설계 문서를 더 일찍 공유하기", "추정 시간 보수적으로 잡기", "코드 리뷰에서 의견 제시 빈도 늘리기"
