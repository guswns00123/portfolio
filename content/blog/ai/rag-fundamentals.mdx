## 들어가며

LLM(GPT, Claude 등)을 직접 써 본 사람이라면 한 번쯤 마주친 한계가 있다.

- "OO 게임의 지난주 패치 노트 알려줘" → 모름 (학습 시점 이후 데이터)
- "우리 회사 DB의 OO 테이블 알려줘" → 모름 (사내 비공개 데이터)
- "이 캐릭터 빌드 추천해줘" → 그럴듯한데 절반은 틀림 (환각)

**최신성, 도메인 데이터, 환각(hallucination)** — 사전 학습된 LLM의 본질적 한계다. 모델 가중치를 다시 학습(fine-tuning)시키지 않고 이 문제를 푸는 표준 패턴이 RAG(Retrieval-Augmented Generation)다.

이 글은 RAG의 핵심 개념을 정리한 뒤, 직접 운영 중인 게임 챗봇 프로젝트에서 **Text-to-SQL의 스키마 Retrieval**에 RAG를 어떻게 적용했고 어떤 단계를 거쳐 pgvector까지 도달했는지 측정 데이터와 함께 풀어본다.

---

## RAG의 전체 흐름

RAG는 두 단계로 구성된다.

**1단계 — Indexing (오프라인)**
원본 정보를 검색 단위로 나눠 임베딩 벡터로 변환하고 벡터 DB에 저장한다.

**2단계 — Retrieval + Generation (온라인)**
사용자 질문을 같은 방식으로 임베딩하고, 벡터 DB에서 의미상 가까운 항목을 검색해 프롬프트에 함께 넣어 LLM에 전달한다.

```
[Indexing]
원본 정보 → 검색 단위 분할 → Embedding → Vector DB

[Retrieval + Generation]
질문 → Embedding → Vector Search → Top-K → Prompt 결합 → LLM → 응답
```

여기서 두 질문이 떠오른다 — "왜 텍스트를 임베딩으로 바꾸지?" / "왜 키워드 검색이 아니지?" 이 둘이 RAG의 핵심을 이해하는 지점이다.

---

## 핵심 개념 1: 임베딩(Embedding)

임베딩은 **텍스트를 의미가 보존된 고정 길이 벡터로 변환**하는 과정이다. OpenAI의 `text-embedding-3-small`은 1,536차원의 float 벡터를 만든다.

같은 의미의 문장은 벡터 공간에서 가까이, 다른 의미의 문장은 멀리 배치된다.

| 문장 A | 문장 B | 의미 유사도 |
|---|---|---|
| "캐릭터 장비 정보" | "프로필에서 착용 중인 무기·방어구" | 가깝다 |
| "캐릭터 장비 정보" | "오늘 점심 메뉴 추천" | 멀다 |

키워드 검색(LIKE, full-text search)이 잡지 못하는 게 이 부분이다. "장비 정보"와 "착용 중인 무기·방어구"는 글자가 거의 겹치지 않지만, 의미상 같은 카테고리다. 임베딩은 이런 **의미적 유사도(semantic similarity)**를 잡아낸다.

이 능력은 임베딩 모델이 학습 과정에서 본 방대한 텍스트로부터 "어떤 단어들이 어떤 맥락에서 함께 등장하는지"를 학습한 결과다. 같은 맥락에서 자주 등장하는 표현은 비슷한 벡터를 갖게 된다.

---

## 핵심 개념 2: 벡터 검색과 유사도

임베딩 두 개가 있을 때 "얼마나 가까운가"를 측정하는 표준 방법이 **코사인 유사도(cosine similarity)**다.

```
cos(A, B) = (A · B) / (|A| × |B|)
```

값의 범위는 −1 ~ 1이며, 1에 가까울수록 두 벡터의 방향이 같다 = 의미가 비슷하다.

벡터 DB에서 "질문과 가장 가까운 Top-K"를 찾는 동작이 곧 **K-Nearest Neighbor 검색**이다. 데이터가 적으면 모든 벡터를 다 비교(brute force)하면 되지만, 수만 건만 넘어도 비현실적이다. 이때 쓰이는 게 **ANN(Approximate Nearest Neighbor)** 인덱스 — HNSW, IVFFlat 같은 구조로 약간의 정확도를 포기하는 대신 검색 속도를 수십~수백 배 끌어올린다.

| 인덱스 | 구조 | 특징 |
|---|---|---|
| IVFFlat | 벡터 공간을 클러스터로 나누고, 가까운 클러스터만 탐색 | 빌드 빠름, 메모리 적음 |
| HNSW | 다층 그래프로 가까운 이웃끼리 연결 | 검색 빠름, 메모리 많음 |

`pgvector`는 PostgreSQL 안에서 벡터 컬럼을 다루게 해 주는 확장이며 두 인덱스를 모두 지원한다. 이 글의 사례에서 핵심 역할을 한다.

## 사례: Text-to-SQL Retrieval에 RAG를 4단계로 진화시킨 과정

내가 운영 중인 [게임 무물봇](https://game.mumulbot.com/) 프로젝트는 자연어 질문을 LLM이 SQL로 변환(Text-to-SQL)해 PostgreSQL에서 데이터를 가져오는 챗봇이다. 여기서 RAG는 흔한 "정적 텍스트 문서 검색"이 아닌, **DB 테이블 스키마 메타데이터를 검색**하는 데 쓰였다.

### 일반 RAG와 다른 점 — 검색 대상이 "텍스트 문서"가 아닌 "스키마"

| 일반 RAG (예: 사내 문서 Q&A) | 이 프로젝트 (Text-to-SQL Retrieval) |
|---|---|
| 검색 대상: 텍스트 문서 (PDF, Wiki, 매뉴얼 등) | 검색 대상: **DB 테이블·컬럼 메타데이터** |
| 단위: 문서를 chunk로 분할 | 단위: **테이블 단위 / 컬럼 단위** (자연 분할 단위가 명확) |
| 검색 결과: 답변 근거 텍스트 | 검색 결과: **SQL 작성에 필요한 스키마 일부** |
| 프롬프트 사용처: 답변 생성 | 프롬프트 사용처: **SQL 생성** |

이 차이 때문에 청킹·overlap 같은 일반 RAG 가이드에서 강조되는 토픽보다, **"어떤 단위로 임베딩할지", "Cold Start 같은 인프라 제약을 어떻게 풀지"**가 훨씬 중요했다.

Text-to-SQL Retrieval 아키텍처는 4단계를 거쳐 진화했다.

### Phase 1. 전체 스키마 주입

가장 단순한 시작점. 사용자의 질문이 어떤 테이블을 건드릴지 모르니, 그냥 **DB 스키마 전체를 프롬프트에 통째로 넣고** LLM에게 SQL을 만들어 달라고 했다.

| 측정 항목 | 값 |
|---|---|
| 총 응답시간 | 2,628ms |
| 소모 토큰 | 5,847 (기준점) |
| 정확도 | 83% |

문제는 명확했다 — **토큰 비용**. 모든 질문에 매번 전체 스키마가 따라붙어, 질문 1개에 6,000 토큰 가까이 흘러갔다. 게다가 스키마가 커질수록 더 악화될 구조였다.

### Phase 2. LLM 자체 판단

"LLM이 스스로 어떤 테이블이 필요한지 판단하게 하자"는 발상. 스키마 전체를 한 번 보여주고 → "이 질문에 필요한 테이블만 이름으로 뽑아 줘" → 그 테이블만 다시 프롬프트에 넣어 SQL 생성. 두 번 호출하지만 두 번째 호출의 컨텍스트는 가벼워진다.

| 측정 항목 | 값 (Phase 1 대비) |
|---|---|
| 총 응답시간 | 3,516ms (+34%) |
| 스키마 선택 소요시간 | 1,613ms |
| 소모 토큰 | 469 (-92%) |
| 정확도 | 83% |

토큰은 92% 줄었지만, **할루시네이션**이 새 문제로 등장했다. LLM이 존재하지 않는 테이블 이름을 만들거나, 비슷해 보이는 다른 테이블로 잘못 매핑했다. 응답 시간도 두 번 호출 영향으로 더 늘어났다.

### Phase 3. In-memory Vector Search

여기서 RAG가 본격적으로 들어왔다. 모든 테이블·컬럼 메타데이터를 임베딩으로 만들어 **애플리케이션 메모리에 올려두고**, 질문이 들어올 때마다 코사인 유사도로 Top-K 테이블만 골라 프롬프트에 주입.

| 측정 항목 | 값 (Phase 1 대비) |
|---|---|
| 총 응답시간 | 2,837ms (+8%) |
| 스키마 선택 소요시간 | 898ms |
| 소모 토큰 | 1,130 (-81%) |
| 정확도 | **100%** |

정확도 100%, 토큰도 81% 감축. LLM의 자체 판단보다 임베딩 기반 의미 검색이 훨씬 안정적이라는 게 측정으로 드러났다. 그런데 또 다른 인프라 문제가 떠올랐다.

서비스가 GCP **Cloud Run**으로 배포되어 있어 컨테이너가 자주 죽고 살아나는데, 컨테이너가 새로 뜰 때마다 임베딩 벡터를 메모리에 다시 올려야 했다. 이 **Cold Start 비용이 약 2,520ms**. 처음 사용자에게 "왜 이렇게 느려?"라는 인상을 줄 수밖에 없었다.

### Phase 4. pgvector — DB-native Vector Search

해결은 **벡터를 RDB 안으로 끌고 들어오는 것**이었다. PostgreSQL 확장인 pgvector를 도입해 임베딩 컬럼과 인덱스를 DB에 직접 두고, 검색을 SQL로 수행.

```sql
CREATE EXTENSION IF NOT EXISTS vector;

-- 스키마 메타데이터 임베딩 테이블 (개념적 형태)
ALTER TABLE schema_metadata
  ADD COLUMN embedding VECTOR(1536);

CREATE INDEX ON schema_metadata
USING hnsw (embedding vector_cosine_ops);

-- 검색
SELECT table_name, description
FROM schema_metadata
ORDER BY embedding <=> $1::vector  -- 코사인 거리
LIMIT 5;
```

| 측정 항목 | 값 (Phase 1 대비) |
|---|---|
| 총 응답시간 | 3,147ms (+20%) |
| 스키마 선택 소요시간 | 1,003ms |
| 소모 토큰 | 923 (**-84%**) |
| 정확도 | **100%** |

총 응답시간 자체는 Phase 3보다 약간 늘었지만(DB 라운드트립이 1회 추가됨), **컨테이너 기동 시 임베딩 로드 단계가 통째로 사라지면서 Cold Start 2,520ms가 0이 됐다**. 첫 사용자가 느끼는 체감 속도가 가장 크게 개선된 변화였다.

부수적인 이득도 있었다. 메타데이터 필터링(예: `WHERE schema = 'lostark'`)과 벡터 유사도 검색을 **단일 쿼리에서** 처리할 수 있게 되어, 별도의 Vector store(Pinecone, Weaviate 등)를 운영할 필요가 없어졌다 — 인프라 단일화.

### 4단계 비교

| Phase | 전략 | 토큰 | 정확도 | 한계 |
|---|---|---|---|---|
| Phase 1 | 전체 스키마 주입 | 5,847 | 83% | 토큰 비용 폭증 |
| Phase 2 | LLM 자체 판단 | 469 (-92%) | 83% | 할루시네이션 |
| Phase 3 | In-memory Vector Search | 1,130 (-81%) | 100% | Cold Start ~2,520ms |
| Phase 4 | **pgvector** (DB-native) | 923 (-84%) | **100%** | — |

상세 아키텍처와 Generator별 LLM 호출 분리(Analysis/SQL/Answer 3단계 구조), Few-shot 프롬프팅으로 SQL 재현성을 잡은 과정 등은 [포트폴리오 문서](https://portfolio-dun-psi-28.vercel.app/portfolio/game-chatbot-pipeline)에 정리해 두었다.

## 회고: 일반 RAG 가이드와 다르게 배운 것

1. **RAG = 텍스트 문서 검색이 전부가 아니다.** 검색 대상은 도메인 맥락에 따라 다양해진다. 이 프로젝트에서는 "테이블 스키마 메타데이터"였고, 이는 SQL 생성이라는 명확한 다음 단계와 결합되어 있었다.

2. **인프라 제약이 RAG 아키텍처를 바꾼다.** 서버리스(Cloud Run)에서 In-memory 방식의 Cold Start가 곧 사용자 첫 인상이라는 걸 측정으로 확인하고서야 pgvector로 옮길 동기가 명확해졌다. 일반 RAG 가이드에서는 거의 다루지 않는 영역이다.

3. **"별도 Vector store" vs "DB-native"의 선택은 운영 단순화 관점에서도 봐야 한다.** 데이터 규모가 폭발적이지 않다면, 메타데이터 필터링과 벡터 검색을 한 DB에서 처리하는 게 운영 부담이 훨씬 작다.

## 정리

| 개념 | 한 줄 요약 |
|---|---|
| RAG | "검색된 정보를 LLM 프롬프트에 합쳐 답을 생성하는 패턴" |
| Embedding | "텍스트를 의미가 보존된 고정 길이 벡터로 변환" |
| Cosine Similarity | "벡터 방향이 얼마나 가까운지 측정. 1에 가까울수록 의미가 비슷" |
| ANN (HNSW / IVFFlat) | "약간의 정확도를 포기하고 벡터 검색을 수십~수백 배 빠르게" |
| pgvector | "PostgreSQL 안에서 벡터 컬럼·인덱스를 다루는 확장. RDB와 Vector를 단일 스토리지로 통합" |

RAG는 결국 **"LLM이 모르는 것을 모르게 두고, 답해야 할 정보만 그때그때 옆에 갖다 주는"** 구조다. 그 "정보"가 무엇인가는 도메인이 정한다 — 사내 문서일 수도, 패치 노트일 수도, 이 글에서처럼 **DB 스키마 자체**일 수도 있다.

다음 글에서는 같은 챗봇에서 **Few-shot 프롬프팅으로 SQL 생성 재현성을 어떻게 잡았는지**, 그리고 **3개 LLM Generator의 토큰·응답시간을 추적하는 자체 모니터링 대시보드를 왜 자체 개발했는지**를 다룰 예정이다.
