## 들어가며

복합 인덱스를 만들 때 가장 많이 받는 질문이 "컬럼 순서를 어떻게 정해야 하나요?"다. 실무에서는 흔히 "카디널리티가 높은 것을 앞에 둔다"는 말을 듣지만, 이 규칙은 반쪽짜리다. 진짜 기준은 **쿼리의 WHERE 절이 어떤 형태인지**와 **B-tree 인덱스가 어떻게 탐색되는지**에 달려 있다.

이 글에서는 복합 인덱스의 내부 구조, Leftmost Prefix 규칙, 카디널리티와 선택도의 차이, 그리고 실무에서 자주 쓰이는 "선택도 10% 이내 규칙"의 실제 근거를 정리한다.

---

## B-tree 인덱스 기초

PostgreSQL의 기본 인덱스는 B-tree다. B-tree는 정렬된 키를 계층적으로 저장하는 자료구조로, 한 번의 키 비교로 절반씩 범위를 줄여 나가기 때문에 탐색 비용이 `O(log N)`이다.

복합 인덱스 `(a, b, c)`는 **"a로 먼저 정렬하고, 같은 a 안에서는 b로 정렬하고, 같은 (a, b) 안에서는 c로 정렬한 단일 정렬 순서"**를 갖는다. 즉 물리적으로는 복합 키 하나로 묶인 단일 B-tree다.

이 구조가 인덱스 사용 가능 여부를 결정한다.

---

## Leftmost Prefix 규칙

B-tree 복합 인덱스는 **왼쪽부터 연속된 컬럼**에 대해서만 효율적으로 사용된다.

인덱스 `(a, b, c)`가 있을 때:

| 쿼리 조건 | 인덱스 사용 |
|---------|-----------|
| `WHERE a = ?` | 사용 |
| `WHERE a = ? AND b = ?` | 사용 |
| `WHERE a = ? AND b = ? AND c = ?` | 사용 |
| `WHERE a = ? AND c = ?` | `a`만 사용, `c`는 필터 |
| `WHERE b = ?` | 사용 못 함 (또는 비효율적 스캔) |
| `WHERE b = ? AND c = ?` | 사용 못 함 |

이게 Leftmost Prefix 규칙이다. `b`만 쓰는 쿼리는 `(a, b, c)` 인덱스를 타지 못한다. B-tree가 "a로 먼저 정렬된" 구조이기 때문에, `a` 조건 없이 `b`만 알아서는 어디를 뒤져야 할지 모르기 때문이다.

### 예외: Index Skip Scan

일부 RDBMS(Oracle 등)는 앞 컬럼의 고유값이 적을 때 앞 컬럼을 건너뛰고 스캔하는 Skip Scan을 지원한다. PostgreSQL은 2024년 현재 공식적으로 Skip Scan을 지원하지 않는다(17 버전 시점까지). 앞 컬럼을 안 쓰는 쿼리가 중요하다면 별도 인덱스를 만드는 게 맞다.

## 등호 조건이 범위 조건보다 앞

복합 인덱스에서 순서를 결정할 때 카디널리티보다 먼저 볼 기준이 있다. **등호(`=`)로 쓰이는 컬럼을 앞에, 범위(`<`, `>`, `BETWEEN`)로 쓰이는 컬럼을 뒤에 둔다.**

이유는 B-tree 탐색 방식에 있다. 등호 조건은 인덱스에서 정확히 한 지점으로 내려가지만, 범위 조건은 "여기부터 저기까지"를 스캔한다. 범위 조건 뒤의 컬럼은 **정렬 순서가 깨지므로** 인덱스로 더 좁히지 못한다.

예시:

```sql
-- 쿼리
SELECT *
FROM orders
WHERE region_code = 'KR'
  AND created_at BETWEEN '2026-01-01' AND '2026-01-31'
  AND status = 'COMPLETED';

-- 나쁜 인덱스: (created_at, region_code, status)
-- created_at이 범위라 뒤 컬럼을 좁히지 못함

-- 좋은 인덱스: (region_code, status, created_at)
-- 등호 → 등호 → 범위 순서
```

`region_code = 'KR'`와 `status = 'COMPLETED'`로 범위를 좁힌 뒤, 마지막에 `created_at`으로 연속 구간 스캔을 한다. 이 순서가 가장 적은 페이지를 읽는다.

---

## 카디널리티 vs 선택도

"카디널리티 높은 것부터"라는 말은 자주 듣지만, 엄밀히 말하면 **선택도(selectivity)**가 기준이다. 두 개념은 관련되지만 같지 않다.

### 카디널리티

카디널리티는 **컬럼이 가질 수 있는 고유값의 개수**다.

- `user_id`: 수백만 개 (높음)
- `status` (enum 5개): 5 (낮음)
- `gender`: 2 (아주 낮음)

### 선택도

선택도는 **특정 쿼리 조건이 걸렀을 때 전체 중 얼마가 남는가**를 비율로 나타낸다.

- `status = 'COMPLETED'` → 전체의 80% 남음 → 선택도 0.8
- `user_id = 12345` → 전체의 0.0001% 남음 → 선택도 0.000001

카디널리티가 높으면 대개 등호 조건의 선택도가 낮지만, 언제나 그런 건 아니다. 카디널리티가 낮아도 값의 분포가 편향되지 않으면 유용한 인덱스가 될 수 있다. 반대로 카디널리티가 높아도 특정 값에 데이터가 몰려 있으면 그 값으로 걸러내는 선택도는 나쁠 수 있다.

**PostgreSQL의 Planner는 카디널리티가 아니라 선택도로 판단한다.** `pg_statistic` 뷰에 저장된 히스토그램과 MCV(Most Common Values)를 기반으로 조건이 걸렀을 때 남는 행 수를 추정한다.

## "선택도 10% 이내 규칙"의 실제 근거

실무에서 자주 인용되는 규칙 중 하나가 **"쿼리가 걸러낸 결과가 전체의 5~10% 이내일 때만 인덱스가 효과적"**이라는 것이다. 어디서 온 숫자이고 얼마나 믿을 만한가?

### 왜 이 규칙이 나왔나

인덱스 스캔은 공짜가 아니다. 인덱스를 읽고 → 해당 행이 있는 힙 페이지를 찾아 읽는 두 단계가 필요하다. 힙 페이지는 디스크 상에서 순차적이지 않을 수 있으므로, 많은 행을 읽어야 한다면 **여러 힙 페이지를 랜덤하게 많이 읽게 된다**.

반면 Seq Scan(순차 스캔)은 테이블 전체를 처음부터 끝까지 순차적으로 읽는다. HDD 시대에는 순차 읽기가 랜덤 읽기보다 10배 이상 빨랐기 때문에, 선택도가 낮지 않으면 Seq Scan이 더 빨랐다.

PostgreSQL Planner의 기본 비용 상수는 다음과 같다.

- `seq_page_cost = 1.0`
- `random_page_cost = 4.0`

Planner는 이 비율을 기준으로 비교한다. 전체 행의 10%를 인덱스로 읽으려면 **대략 전체 페이지 수의 10%를 랜덤 읽기**하게 되는데, 이때 예상 비용이 Seq Scan과 비슷해진다. 이게 "10% 규칙"의 수학적 근거다.

### 규칙의 한계

이 규칙은 몇 가지 전제에 의존한다.

- **스토리지가 HDD**라는 전제. SSD에서는 `random_page_cost`를 1.1~1.5로 낮추는 게 일반적이고, 그러면 인덱스가 효과적인 범위가 훨씬 늘어난다.
- **한 행이 한 페이지에 있다**는 전제. 실제로는 한 페이지에 여러 행이 있으므로, 읽어야 할 페이지 수는 행 수보다 적다 (**Correlation**이 높을수록 더 적어진다).
- **Index-Only Scan이 아니라는 전제**. Index-Only Scan은 힙 방문이 거의 없으므로 선택도가 훨씬 높아도 유리하다.

SSD 환경에서는 "10% 이내"보다 **"20~30% 이내"**까지도 인덱스가 효과적인 경우가 많다. 절대 기준이 아니라 **대략의 감각**으로 받아들이는 게 맞다.

## Index-Only Scan과 Covering Index

PostgreSQL은 **인덱스에 조회 대상 컬럼이 모두 포함**되어 있으면 힙 페이지를 방문하지 않고 인덱스만으로 결과를 만들 수 있다. 이를 Index-Only Scan이라 한다.

```sql
-- 쿼리
SELECT name FROM users WHERE email = 'abc@x.com';

-- 일반 인덱스
CREATE INDEX idx_users_email ON users (email);
-- → 인덱스로 email 찾고 → 힙에서 name 읽음 (Index Scan)

-- Covering 인덱스 (PostgreSQL 11+)
CREATE INDEX idx_users_email_name ON users (email) INCLUDE (name);
-- → 인덱스만으로 name까지 해결 (Index-Only Scan)
```

`INCLUDE` 절은 해당 컬럼을 정렬 키로 쓰지 않고 인덱스 리프 페이지에만 저장한다. 정렬 키로 안 쓰이니 Leftmost Prefix에 영향을 안 주고, 쿼리에서 참조만 하는 컬럼을 넣기에 적합하다.

Index-Only Scan이 실제로 일어나려면 PostgreSQL의 **Visibility Map**이 최신 상태여야 한다. `VACUUM`이 Visibility Map을 갱신하므로, autovacuum이 제대로 돌지 않는 테이블에서는 Index-Only Scan의 효과가 떨어질 수 있다.

## 실무에서의 순서 결정 체크리스트

복합 인덱스 컬럼 순서를 정할 때 다음 순서로 검토하면 된다.

1. **등호(`=`) 조건 컬럼을 먼저**, 범위(`<`, `>`, `BETWEEN`) 조건 컬럼을 뒤에.
2. 등호 조건 컬럼끼리는 **선택도가 낮은(결과를 더 많이 걸러내는) 컬럼을 앞에**.
3. `ORDER BY` 컬럼이 있으면 **정렬 순서와 일치하는 순서**로 두면 Sort 생략 가능.
4. `SELECT` 절에서 자주 조회하는 컬럼은 `INCLUDE`로 추가해 Index-Only Scan 유도.
5. 너무 많은 인덱스를 만들지 말 것 — INSERT/UPDATE 비용이 선형적으로 증가하고, Planner 선택도 느려진다.

## 실행 계획으로 확인하는 법

인덱스 선택이 제대로 됐는지는 `EXPLAIN (ANALYZE, BUFFERS)`로 확인한다.

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders
WHERE region_code = 'KR' AND status = 'COMPLETED'
  AND created_at BETWEEN '2026-01-01' AND '2026-01-31';
```

볼 것들:

- `Index Scan` vs `Bitmap Index Scan` vs `Seq Scan` — 어떤 스캔이 선택됐는지
- `Index Cond`와 `Filter` — 인덱스로 좁힌 조건과 나머지 조건이 분리돼 나옴
- `Rows Removed by Filter` — 인덱스로 못 걸러내고 Filter로 버린 행 수
- `Buffers: shared hit/read` — 실제 페이지 접근량

`Rows Removed by Filter`가 크면 인덱스가 조건을 충분히 좁히지 못한 것이다. 이 경우 컬럼을 추가하거나 순서를 바꿀 지점을 찾을 수 있다.

---

## 요약

- 복합 인덱스는 **"왼쪽부터 연속된 컬럼"**에 대해서만 효율적이다 (Leftmost Prefix).
- 순서 결정 첫 번째 기준은 **등호 먼저, 범위 뒤**. 두 번째가 선택도.
- 카디널리티와 선택도는 다르다. Planner는 선택도로 판단하므로 `pg_statistic`의 분포가 중요하다.
- "선택도 10% 이내" 규칙은 HDD 시대의 경험칙이다. SSD 환경에서는 `random_page_cost`를 낮춰 실제 비용을 반영해야 한다.
- Index-Only Scan을 유도하려면 `INCLUDE`로 Covering Index를 만들고, autovacuum이 Visibility Map을 갱신하도록 유지.
- 인덱스 효과는 항상 `EXPLAIN (ANALYZE, BUFFERS)`로 검증할 것. 추측보다 측정이 먼저.
