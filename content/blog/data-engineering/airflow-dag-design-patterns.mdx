## 들어가며

토이 프로젝트에서 Airflow를 처음 쓸 때는 DAG 하나에 Task 몇 개면 충분했다. 하지만 실무에서 15개 이상의 DAG를 운영하면서 깨달은 건, **DAG의 구조 설계가 파이프라인의 유지보수성과 운영 안정성을 결정한다**는 것이었다.

이 글에서는 DAG를 설계할 때 반복적으로 적용하게 된 패턴 세 가지를 정리해 볼 것이다.

---

## 1. Task 단일 책임 원칙 — 하나의 Task에는 하나의 일만

### 문제: 하나의 Task에 모든 것을 넣는 경우

Airflow를 처음 쓸 때 흔히 빠지는 함정이 있다. "어차피 순서대로 실행되니까" 하고 Extract, Transform, Load를 하나의 Task에 몰아넣는 것이다.

```python
# 안티패턴: 하나의 Task에 여러 책임
@task
def process_all_tables():
    # 10개 테이블 데이터 추출
    for table in tables:
        data = extract(table)
        transformed = transform(data)
        load(transformed)
```

내가 경험했던 소규모 프로젝트에서는 문제가 없었다. 하지만 관리할 테이블이 수십 개로 늘어나면 이런 문제가 생긴다:

- **디버깅 불가**: 3번째 테이블에서 에러가 나도, Airflow UI에는 `process_all_tables` 하나가 실패한 것으로만 보인다. 어떤 테이블에서 실패했는지 로그를 직접 뒤져야 한다.
- **재시도 낭비**: 7번째 테이블에서 실패하면, 이미 성공한 1~6번 테이블까지 처음부터 다시 실행된다.
- **팀원이 읽을 수 없다**: Airflow UI만 보고는 이 DAG가 뭘 하는지 파악이 안 된다.

### 해결: Task를 역할별로 쪼개고, 수많은 테이블은 '동적 할당'으로 처리

처음에는 단순히 테이블별로 `extract_users`, `load_users` 식으로 Task를 분리해 보려 했다. 하지만 실무에서 다루는 원천 테이블은 수십, 수백 개 단위였고, 이를 일일이 하드코딩하는 건 유지보수 지옥으로 가는 길이었다.

게다가 데이터를 가져올 때는 단순히 '추출'과 '적재'만 해서는 안 된다. 제대로 들어왔는지 확인하는 '검수' 과정이 필수다. 그래서 나는 테이블 단위로 파이프라인을 분리하되, **Extract → Load → Validate** 3단계로 Task의 책임을 명확히 나누고, Airflow의 [Dynamic Task Mapping](https://airflow.apache.org/docs/apache-airflow/stable/authoring-and-scheduling/dynamic-task-mapping.html) 기능을 활용해 수많은 테이블을 동적으로 처리하도록 설계했다.

```python
@task
def extract_data(table_name: str):
    return fetch_from_source(table_name)

@task
def load_data(data, table_name: str):
    return insert_into_target(table_name, data)

@task
def validate_data(table_name: str):
    # 적재 검수 로직 (건수 매칭, NULL 체크 등)
    check_quality(table_name)

# 관리해야 할 원천 테이블 목록
source_tables = ["users", "orders", "products", "payments", ...]

# 동적 할당(Dynamic Mapping)을 통한 파이프라인 구성
# 테이블 목록을 순회하며 추출 -> 적재 -> 검수 파이프라인을 알아서 n벌 찍어낸다.
extracted = extract_data.expand(table_name=source_tables)
loaded = load_data.partial().expand(data=extracted, table_name=source_tables)

# 적재가 끝난 이후에 검수가 실행되도록 의존성(>>) 부여
loaded >> validate_data.expand(table_name=source_tables)
```

이렇게 구성하면 원천 테이블이 100개로 늘어나도 코드는 단 한 줄도 길어지지 않는다. 게다가 Airflow UI 상에서 어떤 테이블의 '적재'가 실패했는지, 아니면 '검수'에서 막혔는지 직관적으로 파악할 수 있어 장애 대응 속도가 비약적으로 빨라졌다.


### 적용 사례: Airflow는 어디까지 일을 해야 할까?

Task 단일 책임 원칙을 적용하며 파이프라인을 다듬다 보니, 자연스럽게 부딪힌 고민이 있었다. 바로 **'Airflow가 데이터 변환이나 연산까지 직접 처리하는 게 맞을까?'** 하는 점이다.

원천 데이터를 가져온 후, 예측 모델에 사용할 위험도 결과 데이터를 생성하는 복잡한 파이프라인도 운영하고 있었다. 초기에는 데이터 변환 로직을 Airflow Task 내부에 Python(Pandas 등) 코드로 길게 짜넣기도 했다. 하지만 다뤄야 할 데이터가 커질수록 Airflow Worker에 메모리 부하가 오고, 에러가 났을 때 인프라 문제인지 코드 문제인지 파악하기가 어려워졌다.

정답은 없겠지만, 이 과정을 겪으며 나는 "Airflow는 철저히 오케스트레이터 역할만 수행하게 만든다"는 나름의 기준을 세웠다. 무거운 데이터 변환이나 연산 로직은 전부 DB 내부의 프로시저로 밀어 넣고, Airflow는 단순히 그 프로시저를 실행만 하도록 구조를 바꾼 것이다.

```python
# Airflow Task는 그저 DB의 프로시저를 '트리거' 할 뿐이다.
@task
def run_risk_prediction_procedure():
    # 복잡한 연산은 DB가 하고, Airflow는 실행만 시킨다.
    execute_sql("CALL sp_generate_risk_prediction();")
```

이렇게 역할을 분리하고 나니 얻는 이점이 명확했다. 파이프라인이 돌다가 에러가 나면 "Task가 실패했으니 100% 프로시저(DB 내부 로직) 쪽에 문제가 생겼다"라고 장애 지점을 즉시 격리할 수 있게 되었다. Airflow는 파이프라인의 흐름과 의존성 관리라는 본연의 역할에 집중하고, 무거운 연산은 컴퓨팅 자원이 넉넉한 DB가 처리하게 됨으로써 운영 안정성이 훨씬 높아졌다.

## 2. TaskGroup과 @task 데코레이터

### 문제: Task를 잘게 쪼개면 DAG가 스파게티가 된다

앞서 말한 Task 단일 책임 원칙을 따르면 Task 수가 급격히 늘어난다. 내 경우, 실무에서 '전국 17개 시도별 데이터 산출 파이프라인'을 구축할 때 이 문제에 정면으로 부딪혔다.

시도 하나당 Extract → Transform → Validate → Load라는 4개의 Task가 필요하다. 17개 시도를 이 원칙대로 쪼개면 Task만 68개가 나온다. 이때 문제는 **나만 이 DAG를 보는 게 아니라는 것**이다. 운영 환경에서 68개의 Task가 평면적(flat)으로 나열되어 있으면, 에러가 났을 때 '어떤 지역 데이터 적재에서 터진 건지, 어떤 지역 데이터 추출에서 터진 건지' 직관적으로 파악하기가 불가능에 가깝다.

### 해결: @task_group 데코레이터로 논리적 그룹 만들기

이 복잡도를 해결하기 위해 도입한 것이 바로 TaskFlow API의 `@task_group`이다. 나는 파이프라인 코드를 작성할 때 가독성을 위해 전통적인 PythonOperator 대신 `@task` 데코레이터를 적극 활용하고 있는데, 그룹화 역시 데코레이터를 통해 아주 직관적으로 구현할 수 있었다.

```python
from airflow.decorators import dag, task, task_group
import pendulum

# 전국 17개 시도 리스트
regions = ["seoul", "gyeonggi", "incheon", "busan", "jeju", ...]

@dag(
    dag_id="regional_data_pipeline",
    schedule="@daily",
    start_date=pendulum.datetime(2025, 1, 1, tz="Asia/Seoul"),
    catchup=False,
)
def regional_data_pipeline():

    for region in regions:
        # 17개 시도별로 UI에서 접고 펼 수 있는 논리적 그룹 생성
        @task_group(group_id=f"process_{region}")
        def process_region_group(region_name: str):

            @task(task_id=f"extract_{region_name}")
            def extract_data(r_name: str):
                return fetch_data(r_name)

            @task(task_id=f"transform_{region_name}")
            def transform_data(data):
                return process_logic(data)

            @task(task_id=f"load_{region_name}")
            def load_data(data, r_name: str):
                insert_into_db(r_name, data)

            # 데코레이터의 장점: Python 함수 쓰듯 직관적인 데이터 흐름 정의
            raw_data = extract_data(region_name)
            transformed = transform_data(raw_data)
            load_data(transformed, region_name)

        # 그룹 실행
        process_region_group(region)

regional_data_pipeline()
```

이렇게 짜면 Airflow UI 상에서 68개의 Task가 널브러져 있는 것이 아니라, `[process_seoul]`, `[process_gyeonggi]` 처럼 17개의 깔끔한 박스로 정리된다. 특정 지역에서 장애가 났을 때 해당 박스만 열어보면(Expand) 즉시 디버깅이 가능해서 확실히 직관성이 높아졌다.

## 3. DAG 간 체이닝 — TriggerDagRunOperator

### 문제: 하나의 거대한 DAG는 장애 대응의 재앙이다

만약 "원천 데이터 수집 → 모델별 데이터 산출 → View 생성"이라는 전체 흐름을 하나의 DAG에 전부 넣으면 어떻게 될까?

폐쇄망 환경에서 원격으로 파이프라인을 운영했던 경험을 꺼내보려 한다. 당시 나는 서울에 있었고, 서버는 지방에 있는 폐쇄망 환경이었다. 게다가 다루는 원천 데이터 중에 현업에서 작성한 '수기 데이터'가 많아 데이터 자체의 오타나 누락으로 파이프라인이 터지는 일이 잦았다.

이런 상황에서 모든 작업이 단일 DAG로 묶여있다면 그야말로 재앙이다. 중간에 산출 단계에서 수기 데이터 오류로 에러가 났는데, 수정 후 전체 DAG를 다시 돌리자니 수집부터 다시 시작되어 엄청난 시간이 낭비된다. 그렇다고 특정 Task만 재실행(Clear)하자니 파이프라인이 비대해서 사이드 이펙트를 가늠하기 어려웠다.

### 해결: DAG를 목적별로 분리하고 TriggerDagRunOperator로 연결

이런 운영 경험 덕분에, 나는 파이프라인을 설계할 때 장애 발생 시 **'딱 문제의 구간부터' 재실행할 수 있도록** DAG를 철저히 분리하는 원칙을 세웠다. 그리고 분리된 DAG들의 실행 흐름을 통제하기 위해 [TriggerDagRunOperator](https://airflow.apache.org/docs/apache-airflow/stable/_api/airflow/operators/trigger_dagrun/index.html)를 적극적으로 활용했다.

```python
from airflow.operators.trigger_dagrun import TriggerDagRunOperator
from airflow.decorators import dag, task
import pendulum

@dag(
    dag_id="01_extract_raw_data",
    schedule="@daily",
    start_date=pendulum.datetime(2025, 1, 1, tz="Asia/Seoul"),
    catchup=False,
)
def extract_raw_data():

    @task
    def extract_from_source():
        """원천 DB에서 데이터 추출 후 적재"""
        ...

    # 추출 완료 후 산출 DAG들을 트리거
    trigger_model_a = TriggerDagRunOperator(
        task_id="trigger_model_a",
        trigger_dag_id="02_calculate_model_a",
        wait_for_completion=True,       # 산출이 완료될 때까지 센서처럼 대기
        poke_interval=60,               # 60초마다 상태 확인
        allowed_states=["success"],     # success일 때만 다음으로 진행
        failed_states=["failed"],
    )

    trigger_model_b = TriggerDagRunOperator(
        task_id="trigger_model_b",
        trigger_dag_id="02_calculate_model_b",
        wait_for_completion=True,
        poke_interval=60,
        allowed_states=["success"],
        failed_states=["failed"],
    )

    # 산출 완료 후 View 생성 DAG 트리거
    trigger_view = TriggerDagRunOperator(
        task_id="trigger_create_views",
        trigger_dag_id="03_create_views",
        wait_for_completion=False,      # View 생성은 던져놓고 종료 (fire-and-forget)
    )

    # 의존성 연결
    extract_from_source() >> [trigger_model_a, trigger_model_b] >> trigger_view

extract_raw_data()
```

나는 각 DAG의 ID에 `01_`, `02_`, `03_` 과 같이 실행 순서를 암시하는 접두사를 붙여 네이밍 컨벤션을 맞췄다. 이렇게 하면 Airflow UI에서 이름순으로 정렬했을 때 전체 파이프라인의 흐름이 한눈에 들어온다.

```
01_extract_raw_data        (수집)
02_calculate_model_a       (산출 A)
02_calculate_model_b       (산출 B)
03_create_views            (View 생성)
```

이 구조를 도입하고 나서 폐쇄망 서버에서 에러가 터져도 당황하지 않게 되었다. 수집(01)은 정상적으로 끝났는데 산출(02)에서 수기 데이터 에러가 났다면, 데이터를 수정한 뒤 **산출(02) DAG만 독립적으로 다시 실행하면 되기 때문이다.**

### 이 구조의 장점

| 항목 | 단일 DAG | DAG 분리 + Trigger |
|------|---------|-------------------|
| 장애 범위 | 전체 DAG 실패 | 해당 단계 DAG만 실패 |
| 재실행 | 처음부터 재실행 | 실패한 DAG만 재실행 |
| 스케줄 유연성 | 단일 스케줄 | DAG별 독립 스케줄 가능 |
| 코드 관리 | 파일 하나에 모든 로직 | 목적별 파일 분리 |

## 정리

| 패턴 | 핵심 | 적용 기준 |
|------|------|----------|
| Task 단일 책임 | 하나의 Task에 하나의 일 | "이 Task가 실패하면 어디서부터 재시도하고 싶은가?" |
| TaskGroup + @task | 논리적 그룹화 + 가독성 | "DAG를 처음 보는 사람이 흐름을 파악할 수 있는가?" |
| TriggerDagRunOperator | DAG를 목적별 분리 후 체이닝 | "이 단계들의 스케줄이나 장애 범위가 독립적인가?" |

다음 글에서는 Airflow가 외부 시스템(S3, 메시지 큐 등)과 연동될 때의 패턴 — **Hook과 Sensor**를 다룬다.
