## 들어가며

같은 관계형 DB를 쓰더라도 **목적이 트랜잭션 처리(OLTP)인지 분석(OLAP)인지에 따라 바람직한 스키마 설계는 정반대가 된다**. 한쪽은 정규화를 선호하고, 다른 쪽은 반정규화를 선호한다.

이 글은 왜 그런 차이가 나는지를 워크로드 특성에서부터 짚어가고, Star schema와 Snowflake schema의 선택 기준, 그리고 도메인별 스키마 분리를 통한 논리적 격리 전략까지 정리한다.

---

## OLTP와 OLAP의 근본적 차이

| 특성 | OLTP | OLAP |
|------|------|------|
| 워크로드 | 작은 트랜잭션 다수 | 큰 쿼리 소수 |
| 읽기 vs 쓰기 | 쓰기 비율 높음 | 거의 읽기 전용 |
| 한 트랜잭션이 건드리는 행 수 | 소수 (1~수십) | 대량 (수천~수억) |
| 동시성 | 매우 높음 | 상대적으로 낮음 |
| 응답시간 요구 | 밀리초 | 초~분 |
| 대표 쿼리 | "주문 ID 123의 상태는?" | "지난 분기 카테고리별 매출" |

이 차이가 스키마 설계 원칙을 갈라놓는다.

- OLTP는 **쓰기 일관성과 작은 쓰기의 효율**이 중요
- OLAP는 **대량 스캔과 JOIN 집계의 효율**이 중요

---

## 정규화는 왜 OLTP에 유리한가

정규화는 데이터를 중복 없이 분리해 저장하는 설계 원칙이다. 1NF~3NF, BCNF 등의 단계가 있지만 실무에서 중요한 건 단계 이름이 아니라 **정규화가 주는 이익**이다.

### 1. 쓰기 시 이상(Anomaly) 방지

비정규화된 테이블에 같은 정보가 여러 행에 반복 저장되어 있다고 하자. 고객의 주소 하나를 바꾸려면 모든 관련 행을 전부 갱신해야 한다. 하나라도 놓치면 불일치가 생긴다 — **Update Anomaly**.

정규화된 구조에서는 `customers.address`를 한 번만 바꾸면 된다. `orders`는 `customer_id` FK로 연결돼 있을 뿐이다. 쓰기 비용이 작고 일관성이 보장된다.

### 2. 작은 트랜잭션에 유리

OLTP는 "한 행 찾기 → 수정 → 커밋"의 짧은 트랜잭션이 다수다. 정규화된 테이블은 각각 작고, 필요한 컬럼만 가진다. 한 트랜잭션이 적은 페이지만 건드리므로 락 경합이 적고 버퍼 캐시 효율이 좋다.

### 3. 저장 공간 효율

중복 데이터가 없으니 테이블 크기가 작다. 한 페이지에 더 많은 행이 들어가고, 같은 메모리에 더 많은 테이블이 캐시된다.

### 정규화의 비용

JOIN이 많아진다. 주문 하나의 "고객 이름, 상품 이름, 카테고리 이름"을 모두 가져오려면 4~5개 테이블을 JOIN해야 한다. OLTP에서는 이게 문제가 안 된다 — 한 번에 소수 행만 가져오기 때문이다. 인덱스만 잘 걸려 있으면 모든 JOIN이 Index Lookup으로 끝난다.

---

## 반정규화는 왜 OLAP에 유리한가

OLAP는 반대 상황이다. 수천만 행을 스캔해서 집계하는 쿼리에서 JOIN은 매우 비싸다.

- **JOIN 비용**: Hash Join은 작은 쪽 테이블을 메모리에 올려야 하는데, 분석 쿼리에서는 양쪽 다 크다. Disk Sort-Merge Join까지 가면 더 느려진다.
- **스캔 효율**: 정규화된 구조는 여러 테이블을 왕복해야 한다. 반정규화된 구조는 한 번의 큰 스캔으로 끝난다.
- **I/O 패턴**: OLAP는 순차 읽기가 이상적이다. 중복 데이터가 있어도 순차적으로 읽는 편이, 여러 테이블을 랜덤 하게 왕복하는 것보다 빠를 때가 많다.

### 반정규화의 비용

저장 공간이 커지고, 쓰기가 복잡해지고, 일관성을 스키마 수준에서 보장하기 어렵다. OLAP는 거의 읽기 전용이므로 이 비용을 감수한다. 쓰기는 주기적 ETL이 책임지고, 그 과정에서 일관성을 만들어 둔다.

## Star Schema와 Snowflake Schema

OLAP에서 가장 널리 쓰이는 설계 패턴이다.

### Star Schema

중심에 **Fact 테이블** 하나를 두고, 주변에 **Dimension 테이블**들을 배치한다.

- **Fact 테이블**: 측정 가능한 사건 (`sales_fact` — 매출, 수량 등 숫자)
- **Dimension 테이블**: 사건의 맥락 (`dim_customer`, `dim_product`, `dim_date`)

```
          dim_customer
              |
dim_date — sales_fact — dim_product
              |
          dim_store
```

Fact는 길고 좁은(행 수 많고 컬럼 적은) 테이블이고, Dimension은 짧고 넓은 테이블이다. JOIN은 항상 Fact → Dimension 한 단계로 끝난다.

Dimension 테이블은 **의도적으로 반정규화**된다. `dim_product`에 카테고리 이름, 서브카테고리 이름이 다 들어가 있다. 분석 쿼리 한 번으로 `product_name`, `category_name`을 둘 다 얻으려면 추가 JOIN이 없어야 하기 때문이다.

### Snowflake Schema

Dimension 테이블을 추가로 정규화한 형태. `dim_product`가 `dim_category` 테이블과 분리된다.

```
         dim_category
              ↓
          dim_product
              ↓
        sales_fact
```

장점은 저장 공간 절약과 Dimension 갱신이 간편함. 단점은 쿼리마다 JOIN이 한 단계 더 생긴다는 점이다.

### 언제 Star, 언제 Snowflake

- **Star**: 대부분의 분석 상황. 특히 BI 도구(Tableau, Power BI)는 Star를 전제로 최적화돼 있다.
- **Snowflake**: Dimension이 매우 크고 변경이 빈번한 경우, 또는 저장 공간이 극히 제약적인 경우.

현실에서는 혼합형(일부 Dimension은 정규화, 일부는 반정규화)이 더 흔하다.

## 도메인별 스키마 분리

정규화/반정규화는 테이블 간 관계에 대한 원칙이지만, 더 큰 그림에서는 **스키마(네임스페이스) 수준의 분리**가 설계 결정이 된다.

### PostgreSQL의 스키마

PostgreSQL의 "schema"는 혼동하기 쉬운 용어인데, **테이블/함수/뷰를 묶는 네임스페이스**를 뜻한다.

```sql
CREATE SCHEMA raw;
CREATE SCHEMA mart;

CREATE TABLE raw.orders (...);
CREATE TABLE mart.daily_sales (...);
```

한 DB 내에서 용도별 그룹을 나누는 수단이다.

### 도메인별 분리의 효용

대규모 시스템에서는 같은 DB라도 **논리적으로 분리된 여러 워크로드**가 섞여 있는 경우가 많다. 다음 같은 분리가 자주 쓰인다.

- `raw`: 원천 데이터를 가공 없이 적재
- `staging`: 변환 중간 결과
- `mart` 또는 `analytics`: 최종 분석 테이블
- `api`: 서비스 API가 읽는 View 모음

이 분리가 주는 이익:

- **권한 격리**: `raw` 스키마는 ETL만 쓰기 권한, API 유저는 `api` 스키마만 접근.
- **네이밍 일관성**: 같은 `orders`라도 `raw.orders`와 `mart.orders`는 다른 테이블.
- **마이그레이션 용이**: 특정 스키마만 백업/복제/삭제.
- **인지 부담 감소**: 수백 테이블이 한 네임스페이스에 뒤섞이지 않는다.

### 서버 단위 분리로 가는 지점

스키마 분리만으로 부족할 때가 있다.

- **다른 질병/도메인별로 데이터 규모, 갱신 주기, SLA가 크게 다를 때**
- **대량 분석 쿼리가 OLTP 트래픽에 영향을 줄 때**
- **각 도메인이 독립 팀이 소유하고, 운영 주체가 다를 때**

이런 경우 **DB 서버 자체를 나누는** 게 답이다. 예를 들어 다음 같은 구성이 가능하다.

- 서버 1: 원천 수집 전용 (쓰기 중심)
- 서버 2: 도메인 A의 분석·리포트 (도메인 A 데이터만)
- 서버 3: 도메인 B의 분석·리포트 (도메인 B 데이터만)
- 서버 4: 도메인 C의 분석·리포트 (도메인 C 데이터만)

각 서버는 자기 도메인의 스키마만 가지며, 서버별 리소스와 백업 정책을 독립적으로 운영한다. 도메인 간 쿼리가 필요하면 FDW(Foreign Data Wrapper)나 배치 복제를 쓴다.

### 언제 나눌 것인가

스키마 분리(한 DB 안) → DB 서버 분리로 올라가는 임계점은 보통 다음 중 하나 이상일 때다.

- 단일 서버의 CPU/IO가 포화 상태
- 서로 다른 SLA를 지켜야 함 (한쪽은 실시간, 한쪽은 일배치)
- 장애 격리가 필요 (한쪽이 죽어도 다른쪽은 살아야 함)
- 백업/복구 윈도우가 서로 다름

## JSONB의 자리

PostgreSQL의 `JSONB`는 스키마 설계 결정을 보류할 수 있게 해 주지만, 남용하면 문제가 생긴다.

### JSONB가 적절한 경우

- 필드가 자주 추가/삭제되는 이벤트 로그 (`event_payload` JSONB)
- 외부 API 원본 응답을 보관만 하고 나중에 파싱 (`raw_response` JSONB)
- 스키마가 매우 희소한(sparse) 데이터, 즉 테이블의 대부분 행이 서로 다른 컬럼 집합을 갖는 경우

### JSONB를 쓰지 말아야 할 경우

- 분석 쿼리에서 자주 **필터링·집계하는 값**
- **타입 제약이 중요**한 값 (수량, 가격, 날짜 등)
- 인덱스로 **자주 탐색**하는 값

JSONB는 원자성이 정규화 컬럼보다 약하다. "한 컬럼에 하나의 값"이라는 관계형 모델의 장점(타입 보장, 인덱스, 통계, 플래너 추정)을 모두 희생한다. 분석 대상이 되는 값은 **추출해서 별도 컬럼으로 정규화**하는 것이 대부분 옳다.

## 실전 의사결정 흐름

1. **워크로드부터 확인**: OLTP인가, OLAP인가, 또는 혼합인가.
2. **OLTP에 가까우면 정규화를 기본값으로**. 무결성이 우선.
3. **OLAP에 가까우면 Star Schema 고려**. Dimension은 반정규화.
4. **혼합 워크로드라면 스키마로 분리**. 예: `oltp.orders` + `mart.daily_sales_fact`.
5. **한 서버로 감당 안 되면 서버 분리**. 도메인별 격리.
6. **JSONB는 예외적 용도로만 사용**. 분석 대상 값은 정규화된 컬럼으로 추출.

---

## 요약

- OLTP는 작고 많은 쓰기에 유리한 **정규화된 스키마**가 기본. JOIN은 인덱스로 해결.
- OLAP는 대량 스캔과 집계에 유리한 **반정규화된 스키마**가 기본. Star Schema가 가장 보편적.
- Dimension 테이블은 Star에서는 반정규화, Snowflake에서는 정규화. 대부분의 경우 Star를 먼저 고려.
- 큰 시스템은 스키마(네임스페이스)로 워크로드를 분리하면 권한·인지·운영이 모두 깔끔해진다.
- 도메인별 SLA나 자원 경합이 심하면 **DB 서버 단위로 분리**하는 것이 맞다.
- JSONB는 유연성 도구지, 기본 설계 수단이 아니다. 분석 대상 값은 정규화된 컬럼으로 뽑는다.
